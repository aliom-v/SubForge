# SubForge 链式代理与节点编辑改造计划

## 1. 背景

当前仓库已经具备这些基础能力：

- 可以导入分享链接、完整 Mihomo / sing-box 配置和远程订阅
- 导入时会把 Mihomo `dialer-proxy` / sing-box `detour` 统一映射为 `params.upstreamProxy`
- 可以生成 `mihomo` / `singbox` 托管订阅
- 前端已经能展示链式代理拓扑诊断

但当前实现还有几个明显断层：

1. 前端没有真正可用的节点编辑入口，用户只能导入、删除，不能在界面里稳定修改节点名、启用状态、上游代理等关键字段
2. 链式代理目前只有“导入时识别”和“前端只读诊断”，缺少生成前强校验
3. Mihomo 模板里如果保留了静态 `proxy-groups` 并直接写死节点名，在节点增删改之后容易出现过期引用
4. Mihomo 和 sing-box 都在消费 `params.upstreamProxy`，但校验逻辑没有真正收口到共享核心层

这会导致两个直接问题：

- 页面上看起来“有链路”，但生成时仍可能产出不可直接导入的订阅
- 前端和后端对节点、上游、代理组的理解不完全一致，后续继续加功能时容易再次错位

---

## 2. 改造目标

这次改造只做“内置节点/链式代理编辑器”，不做完整 YAML / JSON 配置编辑器。

目标状态如下：

1. 节点列表支持编辑，而不是只能删除
2. 节点编辑支持修改：
   - 名称
   - 协议
   - 地址
   - 端口
   - 启用状态
   - 凭据和附加参数
   - 上游代理
3. 上游代理通过内置下拉选择，不再要求用户手填 `params.upstreamProxy`
4. 生成前统一执行链式代理校验，阻止缺失上游、循环链路、自引用、禁用上游等错误进入托管输出
5. Mihomo 和 sing-box 都消费同一份链路数据，避免一端能生成、另一端语义不一致
6. Mihomo 输出继续采用“固定骨架 + 动态节点组”的方式，避免静态组直接引用已删除节点

---

## 3. 当前实现基线

本次改造直接基于下面这些现有实现推进：

### 3.1 前端

- 节点页主界面：`apps/web/src/App.tsx`
- 节点 API：`apps/web/src/api.ts`
- Mihomo 链路诊断：`apps/web/src/mihomo-topology.ts`

当前前端已经有：

- 节点导入
- 完整配置导入
- 自动同步源保存与同步
- 链式代理拓扑只读展示

当前前端缺少：

- 节点编辑 UI
- 上游代理选择器
- 生成失败时的结构化链路错误提示

### 3.2 后端与核心层

- 节点持久化：`apps/worker/src/repository.ts`
- 节点接口：`apps/worker/src/index.ts`
- 订阅编译入口：`packages/core/src/compile.ts`
- Mihomo / sing-box 渲染：`packages/core/src/renderers.ts`
- 节点协议字段校验：`packages/core/src/node-protocol-validation.ts`

当前后端已经有：

- `params.upstreamProxy` 基础字段校验
- Mihomo 渲染时把 `upstreamProxy` 输出为 `dialer-proxy`
- sing-box 渲染时把 `upstreamProxy` 输出为 `detour`

当前后端缺少：

- 针对整组节点的链路级校验
- 生成前阻断坏链路
- 统一的链路解析结果模型

## 4. 目标设计

## 4.1 数据模型

第一阶段不改数据库表结构，继续复用：

- `nodes.credentials_json`
- `nodes.params_json`

链式代理继续兼容现有字段：

```json
{
  "upstreamProxy": "上游节点名或稳定代理组名"
}
```

这样做的原因：

- 当前导入链路已经把 `dialer-proxy` / `detour` 统一映射到这个字段
- 现有数据库里已经可能存在这类数据
- 先把编辑、校验、生成打通，比先做字段迁移更重要

第二阶段预留增强方向：

- 在核心层增加更稳定的引用语义，例如 `upstreamProxyRef`
- 逐步减少仅靠“名字匹配”的脆弱性

但这不是本批的阻塞项。

## 4.2 共享链路模型

新增一套共享的“链式代理解析与校验”逻辑，建议放在 `packages/core`。

建议新增模块：

- `packages/core/src/node-chain.ts`

职责：

1. 读取节点的 `params.upstreamProxy`
2. 基于当前节点集合和 Mihomo 模板代理组解析引用
3. 识别这些问题：
   - 上游为空字符串
   - 引用不存在
   - 自引用
   - 节点循环
   - 代理组循环
   - 引用到重名节点
   - 引用到已禁用节点
   - 名称同时匹配节点和代理组
4. 输出共享结果：
   - 每个节点的解析链路
   - 问题列表
   - 是否允许生成

前端和后端都复用这套逻辑，前端不再维护一份独立规则。

## 4.3 节点编辑 UI

前端不做完整模板编辑器，而是做“节点编辑器”。

建议直接在 `apps/web/src/App.tsx` 先补一版内置编辑面板，后续如果体量继续增长，再拆子组件。

第一版编辑器包含：

1. 节点列表操作列新增：
   - `编辑`
   - `启用/禁用`
   - `删除`
2. 编辑区域字段：
   - 名称
   - 协议
   - 地址
   - 端口
   - 启用状态
   - 上游代理下拉
   - `credentials` JSON
   - `params` JSON
3. 上游代理下拉来源：
   - `direct`
   - 当前节点名
   - 当前 Mihomo 骨架里的稳定代理组名

编辑原则：

- `direct` 表示清空 `params.upstreamProxy`
- 其余选项统一写回 `params.upstreamProxy`
- 如果用户改了节点名，保存前要重新对全量节点做链路校验，及时暴露受影响引用

## 4.4 生成策略

Mihomo 继续采用“模板骨架 + 动态节点输出”模式，但进一步收紧约束：

1. `🚀 节点选择` 和 `⚡ 自动选择` 由系统动态生成
2. 其他业务组优先只引用稳定组，而不是把每个节点名平铺到所有业务组
3. 如果模板里保留静态 `proxy-groups`，生成前要清理掉不存在的节点引用

sing-box 继续从同一组节点生成 `outbounds`，并把：

- `params.upstreamProxy` -> `detour`

这样 Mihomo 和 sing-box 都吃同一份链路数据。

## 5. 实施方案

## 5.1 第一阶段：补计划与共享校验

目标：

- 先把“能不能生成”统一收口到核心层

改动：

1. 新增 `packages/core/src/node-chain.ts`
2. 抽出共享链路解析和问题枚举
3. `compileSubscription` 在渲染前执行链路校验
4. 如果链路非法，直接返回错误，不产出订阅内容
5. 错误详情里带上结构化问题列表，方便前端展示

建议错误返回形态：

```json
{
  "code": "VALIDATION_FAILED",
  "message": "node chain validation failed",
  "details": {
    "scope": "node_chain",
    "issues": [
      {
        "nodeId": "node_1",
        "nodeName": "HK 01",
        "kind": "missing_upstream",
        "reference": "Transit A",
        "message": "缺少上游节点或代理组：Transit A"
      }
    ]
  }
}
```

这样可以避免为每种链路问题新增一套散乱错误码，同时足够给前端做精确提示。

## 5.2 第二阶段：补前端节点编辑器

目标：

- 把“导入后可调”变成真实能力

改动：

1. `apps/web/src/App.tsx` 增加当前编辑中的节点状态
2. 节点表格增加 `编辑` / `启用切换`
3. 新增节点编辑表单
4. 接入 `updateNode`
5. 上游代理通过下拉选择，不让用户手输引用名作为主路径
6. 保留 `credentials` / `params` JSON 作为高级兜底入口

前端显示策略：

- 编辑器保存前先做本地链路预校验
- 如果本地就能判断有问题，直接阻止提交
- 如果后端返回结构化链路问题，原样展示

## 5.3 第三阶段：统一前后端链路诊断

目标：

- 前端看的链路与后端真正使用的链路完全一致

改动：

1. 让 `apps/web/src/mihomo-topology.ts` 退化为轻薄适配层，或直接改为复用 `packages/core` 共享逻辑
2. 拓扑表格直接展示共享解析结果
3. 编辑器与生成失败提示共用同一套问题描述

## 5.4 第四阶段：模板和业务组进一步收敛

目标：

- 把“节点动态数据”和“模板业务骨架”边界切干净

改动：

1. 默认 Mihomo 模板固定保留业务组
2. `🚀 节点选择` / `⚡ 自动选择` 动态生成
3. 业务组只引用稳定组
4. 导入完整配置时，如果发现静态组直接依赖节点名，生成前自动做引用净化

---

## 6. 兼容策略

本次改造遵循“先兼容、后收敛”：

1. 继续兼容现有 `params.upstreamProxy`
2. 继续兼容导入时从 `dialer-proxy` / `detour` 写入 `params.upstreamProxy`
3. 不要求现有数据库迁移
4. 不要求用户手动改历史节点

对已有模板的处理：

- 如果模板本身是动态 `proxy-groups`，继续按动态组输出
- 如果模板保留静态 `proxy-groups`，生成前做引用净化
- 如果模板中的业务组直接平铺旧节点名，后续建议在默认骨架里逐步改成只依赖稳定组

---

## 7. 风险与处理

### 7.1 风险：名称引用仍然脆弱

当前 `upstreamProxy` 仍按名字匹配，改节点名会影响引用。

处理：

- 第一阶段先通过强校验把问题显性化
- 第二阶段预留 `upstreamProxyRef` 演进空间

### 7.2 风险：模板中自定义代理组过于复杂

有些导入模板可能存在多层组引用、providers、重名组。

处理：

- 共享解析器只对“可确认问题”做硬拦截
- 对 provider 引用先保留可读诊断，不轻易误伤合法模板

### 7.3 风险：前端一次性塞太多高级字段

如果编辑器直接暴露全部协议字段，界面会很乱。

处理：

- 第一版先解决“能编辑、能改链路、能改名字”
- 协议细节先继续通过 `credentials` / `params` JSON 兜底
- 后续再按协议拆结构化向导

---

## 8. 验收标准

完成后至少满足下面这些验收条件：

1. 节点列表可以编辑节点，不再只有删除
2. 可以在 UI 里修改节点名、启用状态和上游代理
3. 上游代理缺失、循环、自引用、禁用上游时，生成会被阻止
4. Mihomo 和 sing-box 对同一条链路给出一致结果
5. 删除或改名节点后，不会再把过期节点名留在 Mihomo 静态代理组里
6. 测试覆盖至少包含：
   - 缺失上游
   - 自引用
   - 节点循环
   - 禁用上游
   - Mihomo `dialer-proxy`
   - sing-box `detour`

---

本轮按下面顺序落地：

1. 先补这份计划文档
