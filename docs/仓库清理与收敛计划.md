# SubForge 仓库清理与收敛计划

## 1. 文档目的

本文档用于把“接下来还要删什么、哪些先不删、应该按什么顺序删”收成一份正式计划，避免后续继续一边改一边猜。

当前目标不是把仓库盲目删到只剩最小文件数，而是：

1. 继续收敛到当前单用户托管主路径
2. 清掉已经脱离主路径的前端残留、旧命名和重复文档
3. 在不破坏现有托管链路的前提下，再决定哪些后端高级能力是否继续保留

更新时间：`2026-04-01`

---

## 2. 当前基线

### 2.1 当前产品基线

当前仓库的真实产品定位已经收敛为：

- 单用户托管订阅工具
- 主界面围绕三种导入入口展开：
  - 节点文本导入
  - 订阅 URL 解析
  - 导入完整配置
- 三条导入路径最终都回到同一个动作：
  - 使用当前启用节点统一生成托管 URL

当前单用户主链路依赖的底层对象仍包括：

- 用户
- 节点
- 模板
- 用户节点绑定
- 预览缓存
- 公开订阅缓存

这些对象虽然在前端不再作为独立管理页面暴露，但它们仍是当前托管链路的一部分，不能因为“看起来像多用户残留”就直接删除。

### 2.2 当前仓库结构

当前建议按下面这组目录理解项目：

- `apps/web`
  - 当前单用户前端主界面、首次安装向导、管理员登录
- `apps/worker`
  - API、公开订阅输出、Cron、静态资源回退
- `packages/core`
  - 节点解析、订阅编译、同步与缓存失效等核心逻辑
- `packages/shared`
  - 共享类型、错误码、schema、cache key
- `scripts`
  - 初始化、契约检查、smoke、D1 备份恢复等脚本
- `tests`
  - 请求级 / 集成测试与 Web 侧逻辑测试
- `docs`
  - 单用户主线文档 + 内部参考文档
- `.github/workflows`
  - CI、Deploy、D1 Backup、Failure Alert

### 2.3 当前已完成的收敛

截至当前已完成：

- 移除旧的隐藏前端“用户管理”分支和相关面板
- 删除当前前端不再使用的旧 API 封装与路由声明
- 清掉一个完全断引用的旧前端组件：
  - `apps/web/src/template-structure-assistant.tsx`
- 修正 smoke / contract / README / 单用户文档口径
- 归档早期路线图与项目命名说明，主导航不再把历史材料当成当前执行入口
- 修正文档、OpenAPI 与验收清单中残留的旧“后台”表述
- 清理 `apps/web/src` 中未再使用的代理组拓扑辅助导出与一批孤立样式
- 已验证：
  - `XDG_CONFIG_HOME=/tmp WRANGLER_OUTPUT_FILE_DIRECTORY=/tmp npm run ci:verify`

## 3. 清理原则

后续所有清理都按下面四条执行：

### 3.1 先删“死代码”，后删“活能力”

优先删除：

- 零引用文件
- 零引用样式
- 已经不再被当前前端调用的封装
- 明显过时的文档表述

不要直接删除：

- 仍被 Worker、OpenAPI、测试或数据模型依赖的后端能力

### 3.2 先收前端暴露面，再决定后端保留面

当前前端已经不是“完整后台平台”，但后端仍保留了一批高级能力。是否继续保留这些能力，要先明确产品决策，再做删除，不适合直接顺手砍掉。

### 3.3 一次只收一批

每一批应满足：

- 改动边界明确
- 能单独回滚
- `npm run ci:verify` 可独立验证

### 3.4 单用户主链路优先级最高

任何清理都不能破坏：

1. 首次安装向导
2. 管理员登录
3. 三种导入入口
4. 当前启用节点统一生成托管 URL
5. `/s/:token/mihomo` 与 `/s/:token/singbox`

## 4. 待清理对象清单

### 4.1 可以继续做的高置信度清理

这部分属于“可以继续收，但风险较低”的对象。

### A. 文档口径与命名统一

主要问题：

- 少量文档仍把 `apps/web` 或当前界面写成“管理后台”
- 少量文档仍混用“独立管理页面”“前端主界面”“单用户主界面”这几套表述

优先涉及文件：

- `README.md`
- `docs/实施方案.md`
- `docs/排障与常见问题.md`
- `docs/部署指南.md`

处理方式：

- 把当前真实产品界面统一写成“单用户前端主界面”或“节点主界面”
- 只有在明确指代后端 API / 管理员鉴权语义时，才继续使用“后台”这个词

### B. 文档导航与职责边界继续收口

主要问题：

- `docs/` 里的状态文档、计划文档、实施文档之间边界已经比之前清楚，但还缺一份专门面向“继续清理仓库”的执行计划
- 项目接手者仍容易把“已知问题”“实施方案”“清理计划”混在一起看

优先涉及文件：

- `docs/INDEX.md`
- `docs/已知问题与修复计划.md`
- `docs/自动化验证与CI计划.md`
- 本文档

处理方式：

- 维持单用户主线文档不变
- 把“计划 / 状态 / 清理”三类文档入口进一步分清
- 减少重复描述同一件事

### C. 前端残留扫描

主要问题：

- 当前最明显的零引用前端旧组件已经删除
- 但后续仍应继续按“零引用文件 / 零引用样式 / 零引用 helper”做逐轮扫描

重点目录：

- `apps/web/src`
- `apps/web/src/styles.css`

处理方式：

- 每次只删经 `rg`、类型检查和构建确认的真死代码
- 不做“看起来像多余”但未完成依赖核实的批量删除

---

### 4.2 需要先做产品决策的中风险对象

这部分不是“不能删”，而是“删之前必须先决定是否还要保留成高级能力”。

### D. 远程节点一次性导入高级接口

当前对象：

- `POST /api/nodes/import/remote`

当前状态：

- 本轮已下线
- 当前前端主链路统一改为 `node-import/preview` + `nodes/import` 或 `remote_subscription_sources`

本轮已同步处理：

- `apps/worker/src/index.ts`
- `openapi.yaml`
- `scripts/openapi-contract-check.mjs`
- smoke / 文档 / 契约脚本
- README 与计划文档口径

建议：

- 后续不再恢复这条独立手动接口，避免再次和当前主链路并行演化
- 这一项已完成下线；后续只需维持文档与实现口径一致

### E. 规则源 / 同步日志 / 审计日志 / 缓存重建接口

当前对象：

- `/api/rule-sources*`
- `/api/sync-logs`
- `/api/audit-logs`
- `/api/cache/rebuild`

当前状态：

- 当前单用户前端不再把这些对象作为独立页面暴露
- Worker、OpenAPI、测试、ER 图、运维文档仍完整覆盖
- 这些能力仍属于当前仓库的真实后端能力，而不是单纯的“前端残留”

如果决定删除，影响面会很大：

- API 契约
- OpenAPI
- Worker 路由
- repository / service 层
- 多组测试
- 架构图 / API 参考 / 运维文档

建议：

- 先决定“保留为内部高级能力”还是“彻底从产品范围中移除”
- 在没有明确产品决策前，先不做删除

### F. 多用户形态相关的底层对象命名

当前对象：

- `users`
- `templates`
- `user_node_map`
- `preview`

当前状态：

- 名字仍带多用户平台痕迹
- 但当前托管链路仍直接依赖这些对象

建议：

- 不做“为了看起来像单用户”而进行数据库或核心对象重命名
- 如未来真要重构，应作为独立迁移项目处理，不并入普通清理批次

---

### 4.3 明确先不删的对象

下面这些当前明确不应作为“清理目标”：

- `/api/users`
- `/api/users/:id/nodes`
- `/api/templates`
- `/api/preview/:userId/:target`
- 自动托管用户与模板的内部维护逻辑
- 公开订阅 `/s/:token/:target`
- 与缓存正确性相关的 preview/subscription 逻辑

原因很简单：

- 它们仍直接服务当前单用户托管输出
- 不是“旧后台功能”，而是当前主链路的内部实现

## 5. 分批执行计划

### 5.1 Batch 1: 文档与入口整理（已完成）

目标：

- 先把文档入口、项目结构说明、命名口径完全理顺

范围：

- `README.md`
- `docs/INDEX.md`
- `docs/实施方案.md`
- 部分单用户主线文档的用词统一

动作：

- 统一“前端主界面 / 节点主界面 / 后端 API”的叫法
- 让项目结构说明和当前真实结构一致
- 减少重复入口和交叉描述

风险：

- 低

验收：

- 文档导航可直接告诉接手者“先看哪几篇”
- 不再把当前产品写成完整多用户管理平台

### 5.2 Batch 2: 前端死代码继续扫描（本轮已完成一轮）

目标：

- 再做一轮 `apps/web` 级别的零引用文件 / helper / 样式收口

范围：

- `apps/web/src`
- `apps/web/src/styles.css`

动作：

- 按引用关系删真死代码
- 同步删对应样式和测试残留

风险：

- 低到中

验收：

- `npm run typecheck`
- `npm run build:web`
- `npm test`

### 5.3 Batch 3: 内部文档合并与归档判断（已完成当前归档判断）

目标：

- 判断哪些“计划类文档”应该继续单独保留，哪些应合并或归档

优先对象：

- `docs/已知问题与修复计划.md`
- `docs/自动化验证与CI计划.md`
- `docs/archive/2026-03-07-subforge-roadmap.md`

动作：

- 保留仍在维护的
- 把已经失去独立价值的计划文档合并进更合适的长期文档

风险：

- 中

验收：

- `docs/INDEX.md` 中每篇文档的职责都能一句话说清

### 5.4 Batch 4: 后端高级能力保留面决策

目标：

- 先做能力清单，再决定删不删高级 API

优先对象：

- `rule-sources*`
- `sync-logs`
- `audit-logs`
- `cache/rebuild`

动作：

- 先列出调用方、测试覆盖、OpenAPI 影响、运维影响
- 做“保留 / 限缩 / 删除”决策
- 只有在决策明确后，才开始真实删除

风险：

- 高

验收：

- 所有删减都能在 OpenAPI、测试、文档、Worker 路由中闭环

### 5.5 Batch 5: 删除后的总体验收

每一批都应至少通过：

- `npm run test:contract`
- `npm run test:smoke`
- `npm test`
- `npm run test:unit`
- `npm run typecheck`
- `npm run build`
- `npm run build:worker`

在受限环境中可使用：

```bash
XDG_CONFIG_HOME=/tmp WRANGLER_OUTPUT_FILE_DIRECTORY=/tmp npm run ci:verify
```

## 6. 建议的执行顺序

建议按下面顺序推进：

1. 先做 Batch 1
2. 再做 Batch 2
3. 然后处理 Batch 3
4. 最后再进入 Batch 4

原因：

- 先把项目说明和边界收清楚
- 再删容易删的前端残留
- 最后才碰真正有后端语义的高级能力

## 8. 当前结论

当前最合适的下一步已经不再是继续收 Batch 1-3，而是进入 Batch 4 的保留面决策：

1. 先列清高级 API 与运维接口的真实调用方、测试面和文档面
2. 再决定哪些保留为内部能力，哪些限缩或删除
3. 只有在产品边界明确后，才继续做真实下线

这样做的好处是：

- 风险最低
- 验证最直接
- 不会把当前单用户托管链路误删坏
