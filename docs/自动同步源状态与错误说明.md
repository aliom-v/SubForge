# SubForge 自动同步源状态与错误说明

## 1. 这页是干什么的

这页只解释自动同步源最常见的状态字段和错误信息。

如果你在节点页看到同步成功、失败、跳过，或者想知道为什么同步后节点没变，优先看这里。

更完整的排障与运维文档：

- `docs/排障与常见问题.md`
- `docs/运维Runbook与告警处理.md`

## 2. 先看哪几个字段

当前最重要的 4 个字段：

- `last_sync_status`
- `last_sync_message`
- `failure_count`
- `last_sync_details_json`

建议顺序：

1. 先看 `last_sync_status`
2. 再看 `last_sync_message`
3. 如果还不清楚，再展开 `last_sync_details_json`
4. 最后看 `failure_count` 判断是不是连续失败

## 3. `last_sync_status` 是什么意思

### `success`

表示这次同步成功，而且节点集合发生了变化。

常见配套消息：

- `subscription updated (N nodes)`

### `skipped`

表示这次同步成功，但解析后的节点集合和上一次相比没有变化。

常见配套消息：

- `subscription unchanged (N nodes)`

这不是报错。

### `failed`

表示这次同步失败，没有形成新的有效结果。

此时要继续看：

- `last_sync_message`
- `last_sync_details_json.reason`
- `last_sync_details_json.errorSummary`

## 4. `last_sync_message` 该怎么读

这是给人先看的摘要。

常见情况：

- `subscription updated (12 nodes)`
  - 同步成功，节点有变化
- `subscription unchanged (12 nodes)`
  - 同步成功，但没有变化
- `upstream content is empty`
  - 上游响应为空
- `no nodes parsed from upstream subscription`
  - 抓到了内容，但没有解析出节点
- 节点链路校验错误消息
  - 说明不是抓取失败，而是新节点里存在坏链路或循环引用

## 5. `failure_count` 代表什么

- 每次 `failed` 会加 `1`
- 同步成功或 `skipped` 后会归零
- 如果你修改了自动同步源的 `sourceUrl`，历史状态和失败计数也会重置

怎么理解：

- 偶发 `1`
  - 通常先看上游是不是临时异常
- 连续上升
  - 说明这条自动同步源已经持续失败，需要排查

## 6. `last_sync_details_json` 里常见字段

当前常见字段包括：

- `sourceUrl`
- `durationMs`
- `upstreamStatus`
- `fetchedBytes`
- `contentType`
- `contentEncoding`
- `lineCount`
- `importedCount`
- `createdCount`
- `updatedCount`
- `unchangedCount`
- `disabledCount`
- `duplicateCount`
- `errorCount`
- `errorSummary`
- `reason`

### 怎么看这些字段

- `upstreamStatus`
  - 看上游是不是 `200`
- `contentType`
  - 看上游返回的是不是你预期的订阅内容，而不是 HTML 登录页
- `importedCount`
  - 看这次一共识别出多少节点
- `createdCount / updatedCount / disabledCount`
  - 看这次到底改了什么
- `duplicateCount`
  - 看导入时去掉了多少重复节点
- `errorCount / errorSummary`
  - 看这次解析阶段是否已经出现错误或告警
- `reason`
  - 看这次失败或跳过的直接原因

## 7. 节点链路错误长什么样

如果失败原因不是抓取，而是链式代理结构有问题，`details` 里通常会带：

- `scope = node_chain`
- `operation`
- `issueCount`
- `issues`

每条 `issue` 常见字段：

- `nodeName`
- `kind`
- `message`
- `reference`
- `chain`
- `upstreamProxy`

这种情况通常不是上游 URL 挂了，而是新导入的节点里出现了：

- 缺失上游代理
- 循环链路
- 无效引用

## 8. 三种最常见场景

### 8.1 成功但没变化

表现：

- `last_sync_status = skipped`
- `last_sync_message = subscription unchanged (...)`

说明：

- 这次同步跑通了
- 只是新结果和旧结果等价

### 8.2 抓到了内容，但没解析出节点

表现：

- `failed`
- `reason` 或 `last_sync_message` 提示没有节点

高频原因：

- 上游返回的是登录页
- 上游内容为空
- 上游内容格式超出当前解析支持范围

### 8.3 节点链路有问题

表现：

- `failed`
- `scope = node_chain`
- `issues` 里能看到具体节点和错误

高频原因：

- 某节点引用了不存在的上游代理
- 修改后形成了循环链路
