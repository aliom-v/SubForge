## 0. Cloudflare UI / Git 导入建议

如果你希望尽量通过 Cloudflare Dashboard 完成部署，当前仓库建议这样使用：

- Worker：通过 Workers 的 Git 导入或 Deploy to Cloudflare 按钮部署
- D1 / KV / Cron：优先交给 Cloudflare 自动 provision；如果你的账号环境没有自动创建，再在 Dashboard 手动补绑
- 管理员：部署后直接通过后台首次安装向导创建
- Web：当前已可随 Worker 静态资源一起部署；若你仍想拆开托管，也可以再把 `apps/web` 单独导入到 Pages

如果仓库公开后，可在 README 中启用 Deploy to Cloudflare 按钮。

### 0.1 按钮导入时建议确认的字段

根据 Cloudflare 官方文档：

- 如果仓库 `package.json` 定义了自定义 `build` / `deploy` 脚本，Deploy to Cloudflare 会自动预填这些命令
- 如果仓库声明了 `cloudflare.bindings` 描述，Cloudflare 会在导入界面显示这些说明
- 如果 `wrangler.toml` 中省略资源 ID，并使用 `run_worker_first = true`，需要 **`wrangler@4.45.0+`** 才能在部署时自动 provision 资源并绑定到 Worker

对当前仓库，建议在导入界面确认：

- Build command: `npm run build`
- Deploy command: `npm run deploy`
- Root directory: 仓库根目录
- Runtime secret: `ADMIN_JWT_SECRET`
- Wrangler version: 使用仓库依赖安装出的 `wrangler@4.x`

# SubForge 部署指南

本文档说明如何在本地与 Cloudflare 上启动当前版本的 SubForge。

## 1. 前置条件

- Node.js 20+
- npm 10+
- Cloudflare 账号
- 已安装 `wrangler`

## 2. 安装依赖

```bash
npm install
```

## 3. 首次安装与本地数据库

### 3.1 执行 migration

```bash
npx wrangler d1 execute subforge --local --file=migrations/001_init.sql
```

### 3.2 创建管理员

推荐优先通过后台“首次安装向导”创建首个管理员。

如需手动导入，也可以执行：

```bash
npm run seed:admin -- admin your-password | npx wrangler d1 execute subforge --local --file=-
```

### 3.3 导入 demo 数据

```bash
npm run seed:demo | npx wrangler d1 execute subforge --local --file=-
```

建议先执行零依赖 smoke 校验：

```bash
npm run test:smoke
```

## 4. 启动本地服务

### 4.1 启动 Worker

```bash
npm run dev:worker
```

默认地址：`http://127.0.0.1:8787`

### 4.2 启动 Web

```bash
npm run dev:web
```

默认情况下，Vite 会把 `/api`、`/s`、`/health` 代理到本地 Worker。

如果 Worker 不在默认地址，请设置：

```bash
VITE_API_BASE_URL=http://127.0.0.1:8787
```

## 5. Cloudflare 资源配置

需要准备：

- 一个 D1 数据库绑定 `DB`
- 一个 KV namespace 绑定 `SUB_CACHE`
- 一个 Cron Trigger

当前仓库默认使用“无资源 ID + Worker-first”配置：所有请求先进入 Worker，API/订阅/健康检查由代码处理，其余页面再回退到静态资源；这依赖 **`wrangler@4.45.0+`** 自动 provision 资源并补齐绑定。

如果你的 Cloudflare 账号环境没有自动创建出来，可在 Dashboard 中手动确认或补充以下绑定：

- `DB` → D1
- `SUB_CACHE` → KV

同时确认 `vars`：

- `APP_ENV`
- `SUBSCRIPTION_CACHE_TTL`
- `PREVIEW_CACHE_TTL`
- `SYNC_HTTP_TIMEOUT_MS`

## 6. 部署顺序

1. 在 Dashboard 或 Deploy to Cloudflare 中导入 Worker
2. 执行构建命令 `npm run build`，只生成前端静态资源
3. 执行部署命令 `npm run deploy`，由 Worker 发布阶段完成 D1 migration 与 Worker 部署
4. 若页面出现 `Unexpected token '<'` 之类 JSON 解析错误，先确认当前部署已使用这版 `run_worker_first = true` 配置，再继续排查绑定问题
5. 若 Cloudflare 没自动补齐 D1 / KV，则在 Dashboard 手动绑定 `DB` / `SUB_CACHE` 后重试部署
6. 打开后台，完成首次安装向导创建管理员
7. 如有需要再导入 demo 数据
8. 如需单独检查前端构建产物，可再执行一次 `npm run build:web`

Worker 部署脚本：

```bash
npm run deploy
```

仅执行远端 migration：

```bash
npm run db:migrations:apply
```

Web 构建：

```bash
npm run build
```

仅前端构建：

```bash
npm run build:web
```

## 7. 当前支持的功能

- 首次安装向导 / 管理员登录 / 会话校验
- 管理后台随 Worker 静态资源一起发布
- 用户 / 节点 / 模板 / 规则源创建与编辑
- 用户节点绑定
- 订阅预览
- 公开订阅拉取
- 规则源手动同步与 Cron 同步
- 同步日志与审计日志
- 审计日志联表管理员名称，并记录请求上下文
- 预览缓存与公开订阅缓存分离

## 8. 缓存与同步说明

- 公开订阅缓存使用 `SUBSCRIPTION_CACHE_TTL`
- 预览缓存使用 `PREVIEW_CACHE_TTL`
- 用户资料、token、绑定变更会失效对应用户缓存
- 节点更新 / 删除会失效受影响用户缓存
- 模板变更只对受影响目标类型做缓存失效
- 规则源同步会对规则做归一化、去重与内容哈希比对
- 同步日志会记录结构化 `details_json`，包含格式、耗时、上游状态、规则数与原因
- 若规则内容未变化，同步日志会记录 `skipped`

## 9. 当前限制

- `yaml/json` 同步解析仍为轻量实现，不是完整语法解析器
- 还没有完整的 CI / CD 流程，但已提供首次安装向导与 Cloudflare Git 导入能力
