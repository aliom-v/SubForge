# SubForge 部署指南

## 0. 单人使用最小路径

如果你只是想给自己部署、自己使用，优先按这条最小路径走，不必先看 staging、备份归档、回滚演练这类运维增强内容。

推荐顺序：

1. 本地执行 `npm ci`
2. 本地执行 `npm run init:local`
3. 启动 `npm run dev:worker` 和 `npm run dev:web`
4. 打开前端主界面，完成首次安装向导
5. 在“节点”页按唯一主流程导入节点并生成托管 URL
6. 确认 `/s/:token/mihomo` 或 `/s/:token/singbox` 能被客户端拉取

如果你准备直接部署到 Cloudflare，再继续看下面几节：

- `0.1` Dashboard / Git 导入建议
- `4` 远端部署
- `5` GitHub Actions 自动部署

其余备份、回滚、对象存储归档、值班告警一类内容，都属于后续增强，不是单人使用主线的必做项。

## 0.1 Cloudflare UI / Git 导入建议

如果你希望尽量通过 Cloudflare Dashboard 完成部署，当前仓库建议这样使用：

- Worker：通过 Workers 的 Git 导入或 Deploy to Cloudflare 按钮部署
- D1 / KV / Cron：优先交给 Cloudflare 自动 provision；如果你的账号环境没有自动创建，再在 Dashboard 手动补绑
- 管理员：部署后直接通过前端首次安装向导创建
- Web：当前已可随 Worker 静态资源一起部署；若你仍想拆开托管，也可以再把 `apps/web` 单独导入到 Pages

如果仓库公开后，可在 `README.md` 中启用 Deploy to Cloudflare 按钮。

### 0.2 按钮导入时建议确认的字段

根据 Cloudflare 官方文档，对当前仓库建议确认：

- Build command：`npm run build`
- Deploy command：`npm run deploy`
- Root directory：仓库根目录
- Runtime secret：`ADMIN_JWT_SECRET`
- Wrangler version：使用仓库依赖安装出的 `wrangler@4.x`

如果 `wrangler.toml` 中省略资源 ID，并使用 `run_worker_first = true`，建议直接使用仓库当前依赖里的 **`wrangler@4.77.0`**；`4.45.0+` 才开始支持在部署时自动 provision 资源并绑定到 Worker。

### 0.3 Dashboard 首次部署前确认清单

- 目标应选择 **Workers** 项目；当前仓库不要求单独创建 Pages，因为 `apps/web/dist` 已由 Worker `ASSETS` 托管
- `Root directory` 固定指向仓库根目录；如果只指向 `apps/web` 或 `apps/worker`，构建命令、部署命令和绑定都会错位
- `Build command` 保持 `npm run build`，`Deploy command` 保持 `npm run deploy`
- 如需与仓库内 `wrangler.toml` 保持一致，production Worker 服务名应为 `subforge`，staging 应为 `subforge-staging`
- Node.js 版本建议与 workflow 一致使用 `20`；当前仓库根目录已提交 `.nvmrc` 与 `.node-version` 来固定这个版本，避免 Workers Builds 退回默认 Node 版本
- Runtime Secret 至少配置 `ADMIN_JWT_SECRET`
- 首次部署完成后确认 D1 `DB`、KV `SUB_CACHE` 与 Cron 是否已自动 provision；如果没有，请在 Dashboard 中补绑后再次执行部署

### 0.4 三种推荐部署方式

| 场景 | 推荐方式 | 适用时机 |
| --- | --- | --- |
| Cloudflare Dashboard / Git 导入 | Workers Git import + `npm run build` / `npm run deploy` | 生产常驻部署，尽量少依赖本地 CLI |
| CLI 首次远端初始化 | `npm run init:remote -- --admin-user admin --admin-password your-password` | 首次搭环境，希望把 smoke / build / deploy / 首个管理员写入收敛成一条命令 |
| GitHub Actions | `.github/workflows/deploy.yml` | `main` 自动发布，或需要可审计的流水线部署 |

## 1. 前置条件

- Node.js `20`（推荐；当前支持 `>=20 <25`，`Node.js 25+` 暂不支持）
- npm 10+
- Cloudflare 账号
- 已安装 `wrangler`

## 2. 本地最小启动

### 2.1 安装依赖

```bash
npm ci
```

如果你当前本机是 `Node.js 25+`，请先切回仓库固定的 `20`，或其他受支持的 LTS 版本；当前仓库会在安装前直接拒绝不受支持的运行时，避免继续掉进 `sharp` 原生依赖构建失败。

### 2.2 一键初始化（推荐）

```bash
npm run init:local
```

这条命令会先执行 `npm run test:smoke`，再应用本地 D1 migrations。

如需在初始化时一并导入首个管理员和 demo 数据：

```bash
npm run init:local -- --admin-user admin --admin-password your-password --with-demo
```

### 2.3 手动执行 migration

```bash
npx wrangler d1 migrations apply DB --local --config ./wrangler.toml
```

### 2.4 启动本地服务

```bash
npm run dev:worker
npm run dev:web
```

默认情况下，Vite 会把 `/api`、`/s`、`/health` 代理到本地 Worker。

如果 Worker 不在默认地址，请设置：

```bash
VITE_API_BASE_URL=http://127.0.0.1:8787
```

### 2.5 如果不走 `init:local`

推荐优先通过前端“首次安装向导”创建首个管理员；只有在你需要手动拆开步骤时，再单独执行下面几条命令。

手动导入管理员：

```bash
npm run seed:admin -- admin your-password | npx wrangler d1 execute subforge --local --file=-
```

手动导入 demo 数据：

```bash
npm run seed:demo | npx wrangler d1 execute subforge --local --file=-
```

建议的最小本地校验：

```bash
npm run test:smoke
npm run test:unit
```

## 3. Cloudflare 资源与变量

需要准备：

- 一个 D1 数据库绑定 `DB`
- 一个 KV namespace 绑定 `SUB_CACHE`
- 一个 Cron Trigger

当前仓库默认使用“显式资源 ID + Worker-first”配置：所有请求先进入 Worker，API / 订阅 / 健康检查由代码处理，其余页面再回退到静态资源；production / staging 的 D1 与 KV 资源 ID 已维护在 `wrangler.toml` 中，避免 GitHub Actions 与本地 CLI 在执行远端 migration / deploy 时找不到目标资源。

如果你的 Cloudflare 账号环境还没建好这些资源，可在 Dashboard 或 Wrangler CLI 中先创建，再把返回的资源 ID 写回 `wrangler.toml`：

- `DB` → D1
- `SUB_CACHE` → KV

同时确认 `vars`：

- `APP_ENV`
- `ADMIN_LOGIN_RATE_LIMIT_WINDOW_SEC`
- `ADMIN_LOGIN_RATE_LIMIT_MAX_ATTEMPTS`
- `SUBSCRIPTION_CACHE_TTL`
- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS`
- `PREVIEW_CACHE_TTL`
- `SYNC_HTTP_TIMEOUT_MS`

### 3.1 绑定 / Secret / 调度对照

| 名称 | 类型 | 在哪配置 | 用途 | 首次部署要确认 |
| --- | --- | --- | --- | --- |
| `DB` | D1 绑定 | `wrangler.toml` + Dashboard | 存储用户、节点、模板、规则与日志 | migration 能成功执行，前端主界面中的核心资源能正常读写 |
| `SUB_CACHE` | KV 绑定 | `wrangler.toml` + Dashboard | 存储预览与公开订阅缓存 | `/api/preview` 与 `/s/:token/:target` 能正常 miss / hit |
| `ASSETS` | Worker 静态资源绑定 | `[assets]` 自动生成 | 托管 `apps/web/dist` | 打开 `/` 能进入 SPA，而不是 404 或默认静态页 |
| `ADMIN_JWT_SECRET` | Runtime Secret | Dashboard 或 Wrangler Action | 管理员会话签名 | 登录后 Bearer token 能持续校验通过 |
| `0 * * * *` | Cron Trigger | `[triggers]` + Dashboard | 每小时规则源同步 | Dashboard 中能看到 Trigger 已生效 |


如果你在 Dashboard 中覆盖 `vars`，建议优先与 `wrangler.toml` 保持一致，尤其是 `SYNC_HTTP_TIMEOUT_MS`、缓存 TTL、管理员登录限流与公开订阅频控相关字段。

### 3.2 高级数据维护入口

如果你只是自己部署自己用，这一节通常可以先跳过。

只有在下面这些场景，再进入高级运维链路：

- production 发版前要做数据备份
- 需要恢复、回滚或离线演练
- 需要把 D1 备份继续同步到对象存储
- 需要把部署接到 GitHub Actions

相关内容已单独收敛到：

- `docs/高级部署与数据维护.md`

最常用的高级命令只有这几条：

```bash
npm run backup:d1
npm run backup:d1:staging
npm run d1:restore:drill -- --file ./backups/d1/<backup.sql>
npm run ci:verify
```

## 4. 部署顺序

1. 在 Dashboard 或 Deploy to Cloudflare 中导入 Worker
2. 执行构建命令 `npm run build`，只生成前端静态资源
3. 执行部署命令 `npm run deploy`，由远端 migration 与 Worker 发布共同完成 D1 schema 更新和 Worker 部署；当前 `deploy` / `deploy:staging` / `deploy:worker*` 在真正发版前都会先刷新 `apps/web/dist`
4. 若目标环境缺少 D1 / KV，请先在 Dashboard 或 Wrangler CLI 创建资源，并把 `database_id` / KV `id` 回填到 `wrangler.toml` 后再重试部署
5. 打开前端主界面，完成首次安装向导创建管理员
6. 如有需要再导入 demo 数据
7. 如需单独检查前端构建产物，可再执行一次 `npm run build:web`

如需通过 CLI 收敛为单条命令，可在完成 Cloudflare Secrets 与绑定配置后执行：

```bash
npm run init:remote -- --admin-user admin --admin-password your-password
```

这条命令会串行执行 `npm run test:smoke`、`npm run build`、`npm run deploy`，并把首个管理员直接写入远端 D1。

常用命令：

```bash
npm run init:local
npm run init:remote -- --admin-user admin --admin-password your-password
npm run build
npm run deploy
npm run db:migrations:apply
npm run build:web
```

如果页面出现 `Unexpected token '<'` 之类 JSON 解析错误，先确认当前部署已使用这版 `run_worker_first = true` 配置，再继续排查绑定问题；详细排查请看 `docs/排障与常见问题.md`。

## 5. 部署后最小验收

建议至少验证以下 5 点：

1. `GET /health` 返回 `200`
2. `GET /api/setup/status` 返回 JSON
3. 前端主界面能正常打开，并可完成首次安装向导或管理员登录
4. 任意一个公开订阅 token 能返回订阅原文
5. 如果你已导入规则源，最近同步日志里的 `details_json` 应能看到格式、阶段、错误码、耗时等结构化信息

如果需要更完整的巡检、告警和事件响应流程，请看 `docs/运维Runbook与告警处理.md`。

### 5.1 推荐的首轮验收顺序

建议按下面顺序完成首轮验收：

1. 打开 `/`，确认能进入前端 SPA，而不是 Worker 默认 404 页面
2. 请求 `/health` 与 `/api/setup/status`，确认健康检查和 JSON API 都正常
3. 如实例尚未初始化，在首次安装向导创建管理员；如已 seed，则直接登录管理员入口
4. 如果已导入 demo 数据或已有真实用户，验证一个公开订阅地址，例如 `/s/:token/mihomo` 或 `/s/:token/singbox`
5. 手动同步一个规则源，确认同步详情中能看到 `stage`、`errorCode`、`operatorHint`、`retryable`；失败时还能看到 `supportedShapes` 与 `contentPreview`
6. 在节点页执行一次节点 JSON 批量导入，确认节点数量、审计日志与后续预览数据都会更新
7. 如启用了自动同步源，再执行一次自动同步，确认节点数、同步日志、审计日志以及后续预览 / 公开订阅都会更新
8. 手动点击“重建订阅缓存”，再重新访问预览或公开订阅，确认最新内容能重新生成

### 5.2 部署后首轮排障观察点

- `/api/*` 返回 HTML 或前端出现 `Unexpected token '<'`：优先检查 `wrangler.toml` 中 `[assets]`、`directory = "./apps/web/dist"` 与 `run_worker_first = true`；当前部署脚本会自动重建前端，如问题仍在，重点排查是否发错环境、浏览器缓存未刷新，或 Cloudflare 侧仍指向旧 Worker 版本
- 首页或登录页在部署后仍然是旧界面：先确认 `/` 返回的 HTML 响应头里 `cache-control` 已是 `no-store, max-age=0, must-revalidate`，再排查是否访问了旧环境、旧域名或浏览器仍保留旧标签页
- 前端主界面空白、首次安装向导异常或登录失败：检查 `ADMIN_JWT_SECRET` 是否已配置，以及 D1 `DB` 是否绑定到当前 Worker
- 规则源长期停留在 `failed`：先看“同步状态 / 同步日志”，重点观察 `details_json` 中的 `stage`、`errorCode`、`operatorHint`、`retryable`、`supportedShapes` 与 `contentPreview`
- 自动同步源同步失败：先确认上游是否返回分享链接、Base64 订阅文本或可识别的 YAML / JSON，再看同步日志与审计日志
- Deploy 成功但数据未生效：确认远端 migration 是否真的执行过，可重新执行 `npm run db:migrations:apply`
- Cron 没有按预期触发：确认 Dashboard 中 Trigger 已存在，且当前 Trigger 绑定的是同一个 Worker 服务

## 6. 自动化与高级运维入口

如果你只是自己部署自己用，这一节通常也可以先跳过。

当前仓库已经提供：

- `npm run ci:verify`
- `.github/workflows/ci.yml`
- `.github/workflows/deploy.yml`
- `.github/workflows/d1-backup.yml`
- `.github/workflows/failure-alert.yml`

详细的 GitHub Actions、`workflow_dispatch`、`git_ref`、`release/*`、`FAILURE_WEBHOOK_URL`、`D1_BACKUP_ARCHIVE_PASSPHRASE`、`D1_BACKUP_ARCHIVE_S3_URI`、`D1_BACKUP_ARCHIVE_ENDPOINT_URL`、`D1_BACKUP_ARCHIVE_SSE`、`D1_BACKUP_ARCHIVE_KMS_KEY_ID` 等高级运维细节，统一放到：

- `docs/高级部署与数据维护.md`

```bash
XDG_CONFIG_HOME=/tmp WRANGLER_OUTPUT_FILE_DIRECTORY=/tmp npm run ci:verify
```
