# SubForge 部署指南

本文档说明如何在本地与 Cloudflare 上启动当前版本的 SubForge，并重点写清首次部署、远端 migration、首次初始化和常见排错路径。

## 1. 这版最重要的部署差异

先记住这三个结论：

1. `npm run build` 只负责构建 Web 静态资源
2. `npm run deploy` 才是完整发布命令，它会先执行远端 D1 migration，再发布 Worker
3. `npx wrangler versions upload` 只上传 Worker 版本，不会执行 `wrangler d1 migrations apply`，所以不适合作为首次部署命令

这也是为什么有些环境构建成功了，但打开页面仍会报：

```text
VALIDATION_FAILED: D1_ERROR: no such table: admins
```

出现这个报错时，不是前端没构建好，而是远端 D1 还没有应用 SubForge 的 migration。

如果你不确定这份文档之后应该继续看哪里，先看：

- `docs/文档导航与阅读顺序.md`

如果你部署后还要继续查接口、字段或错误口径，再看：

- `docs/API参考与接口约定.md`
- `docs/节点字段字典.md`
- `docs/常见错误与返回语义.md`
- `docs/发布前检查清单.md`

当前文档默认先把主环境跑通；非生产分支预览不是必需步骤。如果你后续再开 preview 环境，需要自行确保它也绑定了正确的 D1 / KV，并且执行过 migration。

### 1.1 如果页面停在“加载安装状态”

后台首页一开始会先请求：

- `GET /api/setup/status`

页面上对应的提示就是：

- `加载安装状态`
- `正在检查是否已完成首次初始化...`

如果这一步直接返回：

```text
VALIDATION_FAILED: D1_ERROR: no such table: admins
```

含义不是“前端坏了”，而是 Worker 已经起来了，但它访问的那套远端 D1 还没有 `admins` 表。也就是说，问题基本都在 migration 或绑定，而不是 React 构建。

## 2. Cloudflare UI / Git 导入建议

如果你希望尽量通过 Cloudflare Dashboard 完成部署，当前仓库建议这样使用：

- Worker：通过 Workers 的 Git 导入或 Deploy to Cloudflare 按钮部署
- D1 / KV / Cron：优先交给 Cloudflare 自动 provision；如果账号环境没有自动创建，再在 Dashboard 手动补绑
- 管理员：部署后直接通过后台首次安装向导创建
- Web：当前已可随 Worker 静态资源一起部署；若你仍想拆开托管，也可以再把 `apps/web` 单独导入到 Pages

如果仓库公开后，可在 README 中启用 Deploy to Cloudflare 按钮。

### 2.1 按钮导入时建议确认的字段

根据当前仓库脚本与 `wrangler.toml` 配置，导入界面建议确认：

- Build command: `npm run build`
- Deploy command: `npm run deploy`
- Root directory: 仓库根目录
- Runtime secret: `ADMIN_JWT_SECRET`
- Wrangler version: 使用仓库依赖安装出的 `wrangler@4.x`

额外说明：

- 仓库 `package.json` 已定义自定义 `build` / `deploy` 脚本，Deploy to Cloudflare 应该会自动预填
- 仓库声明了 `cloudflare.bindings` 描述，Cloudflare 导入界面会显示绑定用途说明
- `wrangler.toml` 采用“无资源 ID + `run_worker_first = true`”模式，需要 **`wrangler@4.45.0+`** 才能更稳地自动 provision 并绑定资源

### 2.2 非生产分支 / Preview 构建要不要先开

首次部署建议先不要把注意力放到 branch preview 或 non-production build。

原因很简单：

1. 当前最重要的是先确认主环境已经绑定了正确的 `DB` / `SUB_CACHE`
2. 只要 preview 环境没有单独的 D1 / KV 规划，就很容易把“主环境问题”和“预览环境问题”混在一起
3. preview 就算能成功构建，也一样需要 migration；它不是绕过数据库初始化的捷径

建议顺序：

1. 先把主环境用 `npm run deploy` 跑通
2. 确认首次安装向导、管理员登录、模板/节点/用户主链路都正常
3. 再决定是否给 preview 单独配置资源

## 3. 前置条件

- Node.js 20+
- npm 10+
- Cloudflare 账号
- 已安装 `wrangler`

## 4. 安装依赖

```bash
npm install
```

## 5. 首次安装与本地数据库

### 5.1 执行 migration

```bash
npm run db:migrations:apply:local
```

该命令会按顺序应用 `migrations/` 下的本地 D1 migration，包括管理员会话撤销所需的 `002_admin_session_revocation.sql`。

### 5.2 创建管理员

推荐优先通过后台“首次安装向导”创建首个管理员。

如需手动导入，也可以执行：

```bash
npm run seed:admin -- admin your-password | npx wrangler d1 execute subforge --local --file=-
```

### 5.3 导入 demo 数据

```bash
npm run seed:demo | npx wrangler d1 execute subforge --local --file=-
```

建议先执行零依赖 smoke 校验：

```bash
npm run test:smoke
```

## 6. 启动本地服务

### 6.1 启动 Worker

```bash
npm run dev:worker
```

默认地址：`http://127.0.0.1:8787`

### 6.2 启动 Web

```bash
npm run dev:web
```

默认情况下，Vite 会把 `/api`、`/s`、`/health` 代理到本地 Worker。

如果 Worker 不在默认地址，请设置：

```bash
VITE_API_BASE_URL=http://127.0.0.1:8787
```

## 7. Cloudflare 资源配置

需要准备：

- 一个 D1 数据库绑定 `DB`
- 一个 KV namespace 绑定 `SUB_CACHE`
- 一个 Cron Trigger

当前仓库默认使用“无资源 ID + Worker-first”配置：所有请求先进入 Worker，API/订阅/健康检查由代码处理，其余页面再回退到静态资源；这依赖 **`wrangler@4.45.0+`** 自动 provision 资源并补齐绑定。

如果你的 Cloudflare 账号环境没有自动创建出来，可在 Dashboard 中手动确认或补充以下绑定：

- `DB` → D1
- `SUB_CACHE` → KV

同时确认 `vars`：

- `APP_ENV`
- `SUBSCRIPTION_CACHE_TTL`
- `PREVIEW_CACHE_TTL`
- `SYNC_HTTP_TIMEOUT_MS`

## 8. 部署顺序

### 8.1 推荐的首次部署顺序

1. 在 Dashboard 或 Deploy to Cloudflare 中导入 Worker
2. 执行构建命令 `npm run build`，只生成前端静态资源
3. 执行部署命令 `npm run deploy`，由 Worker 发布阶段完成 D1 migration 与 Worker 部署
4. 若页面出现 `Unexpected token '<'` 之类 JSON 解析错误，先确认当前部署已使用这版 `run_worker_first = true` 配置，再继续排查绑定问题
5. 若 Cloudflare 没自动补齐 D1 / KV，则在 Dashboard 手动绑定 `DB` / `SUB_CACHE` 后重试部署
6. 打开后台，完成首次安装向导创建管理员
7. 如有需要再导入 demo 数据
8. 如需单独检查前端构建产物，可再执行一次 `npm run build:web`

### 8.2 这几个命令分别做什么

完整部署：

```bash
npm run deploy
```

只执行远端 migration：

```bash
npm run db:migrations:apply
```

只发布 Worker：

```bash
npm run deploy:worker
```

只上传新版本但不跑 migration 的命令示例：

```bash
npx wrangler versions upload
```

最后这个命令只适合你已经明确知道远端数据库结构是最新时再用。对首次部署或刚加过 migration 的环境，不要拿它替代 `npm run deploy`。

### 8.3 Cloudflare Dashboard 里应该怎么点

如果你只能在 Cloudflare UI 里操作，建议顺序是：

1. 先在 Worker / Git 导入界面确认 Build command=`npm run build`、Deploy command=`npm run deploy`
2. 确认绑定里存在 `DB` 和 `SUB_CACHE`
3. 确认 Secret 里存在 `ADMIN_JWT_SECRET`
4. 等首次部署结束后，直接打开 Worker 域名
5. 页面进入“首次安装向导”后，创建首个管理员

如果部署成功但页面一打开就报 `no such table: admins`，说明部署阶段没有把 migration 跑到远端 D1，继续看下一节排错。

### 8.4 首次部署最常见报错：`no such table: admins`

报错示例：

```text
VALIDATION_FAILED: D1_ERROR: no such table: admins
```

含义：

- Worker 已经成功启动
- 前端资源也已正常返回
- 但 `GET /api/setup/status` 访问远端 D1 时，发现 `admins` 表不存在

通常原因：

1. 部署命令实际用了 `npx wrangler versions upload`
2. 远端 D1 绑定存在，但 migration 从未执行
3. 选错了 D1 数据库或环境，Worker 指向的不是你刚检查的那一套库

优先修复方式：

```bash
npm run db:migrations:apply
```

如果只能通过 Dashboard 修复：

1. 打开 Cloudflare Dashboard
2. 进入 D1
3. 选择绑定给 `DB` 的那个数据库
4. 打开 SQL Console，而不是 Explore data
5. 先执行 `migrations/001_init.sql`
6. 再执行 `migrations/002_admin_session_revocation.sql`

注意：

- `Explore data` 更适合查看表和数据，不适合作为首次 migration 入口
- `001_init.sql` 里已经包含核心表与索引创建，例如 `admins`、`users`、`nodes`、`templates`、`rule_sources`、`audit_logs`
- `002_admin_session_revocation.sql` 会补 `admins.session_not_before` 字段，供服务端会话撤销使用

排查数据库是否正确时，可执行：

```sql
SELECT name FROM sqlite_master WHERE type = 'table' ORDER BY name;
```

正常情况下，至少应看到这些核心表：

- `admins`
- `users`
- `nodes`
- `templates`
- `rule_sources`
- `rule_snapshots`
- `user_node_map`
- `audit_logs`
- `sync_logs`

如果结果只有 `_cf_KV` 或其他内部表，优先检查两件事：

1. 当前选中的数据库是不是 Worker 绑定 `DB` 的那一个
2. 当前数据库是否真的应用过 `migrations/` 下的 SQL

如果你已经在 SQL Console 里执行过 migration，建议立刻再做两步复核：

1. 重新执行一次上面的 `sqlite_master` 查询，确认 `admins` 已经出现
2. 刷新 Worker 域名，再看首页是否仍停在“加载安装状态”

### 8.5 首次进入后台后的最小操作路径

初始化管理员后，建议用下面的顺序验证主链路：

1. 在“模板”页创建一个启用中的默认模板，例如 `mihomo`
2. 在“节点”页手动创建一个带 `credentials` / `params` 的节点
3. 在“用户”页创建一个用户，记录它的 `token`
4. 在“用户”页把刚创建的节点绑定到这个用户
5. 到“预览”页验证 `/api/preview/:userId/:target` 输出
6. 再访问 `/s/:token/mihomo` 或 `/s/:token/singbox` 验证真实订阅

如果你已经能登录后台，但觉得“导入没反应”或“订阅没变化”，通常不是部署问题，而是进入了使用链路问题。下一步优先看：

- `docs/节点管理与订阅使用说明.md`
- `docs/协议支持矩阵与落地计划.md`

节点录入和订阅验证的更细说明见：

- `docs/节点管理与订阅使用说明.md`
- `docs/节点协议示例库.md`
- `docs/导入成功但订阅没变化排障指南.md`

## 9. 当前支持的功能

- 首次安装向导 / 管理员登录 / 会话校验 / 服务端登出撤销
- 管理后台随 Worker 静态资源一起发布
- 用户 / 节点 / 模板 / 规则源创建、编辑与删除
- 用户节点绑定
- 订阅预览
- 公开订阅拉取
- 规则源手动同步与 Cron 同步
- 同步日志与审计日志
- 审计日志联表管理员名称，并记录请求上下文
- 预览缓存与公开订阅缓存分离

## 10. 缓存与同步说明

- 公开订阅缓存使用 `SUBSCRIPTION_CACHE_TTL`
- 预览缓存使用 `PREVIEW_CACHE_TTL`
- 用户资料、token、绑定、删除变更会失效对应用户缓存
- 节点更新 / 删除会失效受影响用户缓存
- 模板创建 / 更新 / 删除 / 默认切换只对受影响目标类型做缓存失效
- 规则源启停 / 删除 / 内容同步变化会失效全部用户缓存
- 规则源同步会对规则做归一化、去重与内容哈希比对
- 同步日志会记录结构化 `details_json`，包含格式、耗时、上游状态、规则数与原因
- 若规则内容未变化，同步日志会记录 `skipped`

## 11. 当前限制

- `yaml/json` 同步解析仍为轻量实现，不是完整语法解析器
- 还没有完整的 CI / CD 流程，但已提供首次安装向导与 Cloudflare Git 导入能力
- 节点当前支持手动录入、`vless://` / `trojan://` / `vmess://` / `ss://` / `hysteria2://` / `hy2://` 分享链接导入、Base64 包装订阅文本自动解包，以及订阅 URL 一次性远程抓取预览导入
- `ss` 已补结构化向导，但当前向导主要覆盖 `cipher` / `password` / `plugin`
- `hysteria2` 已支持常见单端点分享链接导入、远程预览、常见字段结构化向导与协议级字段校验，但多端口 authority 也还没覆盖，导入后建议继续核对 JSON
- 如果节点已经导入成功，但后台预览或真实订阅仍感觉“没变化”，优先检查三件事：是否真正创建了节点、是否完成用户绑定、是否存在启用中的默认模板
- 如果你要判断某协议现在到底支持到“手动录入 / 导入 / 向导 / 输出”的哪一层，请额外查看 `docs/协议支持矩阵与落地计划.md`
