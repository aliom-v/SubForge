# SubForge 部署指南

## 0. Cloudflare UI / Git 导入建议

如果你希望尽量通过 Cloudflare Dashboard 完成部署，当前仓库建议这样使用：

- Worker：通过 Workers 的 Git 导入或 Deploy to Cloudflare 按钮部署
- D1 / KV / Cron：优先交给 Cloudflare 自动 provision；如果你的账号环境没有自动创建，再在 Dashboard 手动补绑
- 管理员：部署后直接通过后台首次安装向导创建
- Web：当前已可随 Worker 静态资源一起部署；若你仍想拆开托管，也可以再把 `apps/web` 单独导入到 Pages

如果仓库公开后，可在 `README.md` 中启用 Deploy to Cloudflare 按钮。

### 0.1 按钮导入时建议确认的字段

根据 Cloudflare 官方文档，对当前仓库建议确认：

- Build command：`npm run build`
- Deploy command：`npm run deploy`
- Root directory：仓库根目录
- Runtime secret：`ADMIN_JWT_SECRET`
- Wrangler version：使用仓库依赖安装出的 `wrangler@4.x`

如果 `wrangler.toml` 中省略资源 ID，并使用 `run_worker_first = true`，需要 **`wrangler@4.45.0+`** 才能在部署时自动 provision 资源并绑定到 Worker。

### 0.2 Dashboard 首次部署前确认清单

- 目标应选择 **Workers** 项目；当前仓库不要求单独创建 Pages，因为 `apps/web/dist` 已由 Worker `ASSETS` 托管
- `Root directory` 固定指向仓库根目录；如果只指向 `apps/web` 或 `apps/worker`，构建命令、部署命令和绑定都会错位
- `Build command` 保持 `npm run build`，`Deploy command` 保持 `npm run deploy`
- Node.js 版本建议与 workflow 一致使用 `20`；当前仓库根目录已提交 `.nvmrc` 与 `.node-version` 来固定这个版本，避免 Workers Builds 退回默认 Node 版本
- Runtime Secret 至少配置 `ADMIN_JWT_SECRET`
- 首次部署完成后确认 D1 `DB`、KV `SUB_CACHE` 与 Cron 是否已自动 provision；如果没有，请在 Dashboard 中补绑后再次执行部署

### 0.3 三种推荐部署方式

| 场景 | 推荐方式 | 适用时机 |
| --- | --- | --- |
| Cloudflare Dashboard / Git 导入 | Workers Git import + `npm run build` / `npm run deploy` | 生产常驻部署，尽量少依赖本地 CLI |
| CLI 首次远端初始化 | `npm run init:remote -- --admin-user admin --admin-password your-password` | 首次搭环境，希望把 smoke / build / deploy / 首个管理员写入收敛成一条命令 |
| GitHub Actions | `.github/workflows/deploy.yml` | `main` 自动发布，或需要可审计的流水线部署 |

## 1. 前置条件

- Node.js 20+
- npm 10+
- Cloudflare 账号
- 已安装 `wrangler`

## 2. 本地最小启动

### 2.1 安装依赖

```bash
npm ci
```

### 2.2 一键初始化（推荐）

```bash
npm run init:local
```

这条命令会先执行 `npm run test:smoke`，再应用本地 D1 migrations。

如需在初始化时一并导入首个管理员和 demo 数据：

```bash
npm run init:local -- --admin-user admin --admin-password your-password --with-demo
```

### 2.3 手动执行 migration

```bash
npx wrangler d1 migrations apply DB --local --config ./wrangler.toml
```

### 2.4 启动本地服务

```bash
npm run dev:worker
npm run dev:web
```

默认情况下，Vite 会把 `/api`、`/s`、`/health` 代理到本地 Worker。

如果 Worker 不在默认地址，请设置：

```bash
VITE_API_BASE_URL=http://127.0.0.1:8787
```

### 2.5 如果不走 `init:local`

推荐优先通过后台“首次安装向导”创建首个管理员；只有在你需要手动拆开步骤时，再单独执行下面几条命令。

手动导入管理员：

```bash
npm run seed:admin -- admin your-password | npx wrangler d1 execute subforge --local --file=-
```

手动导入 demo 数据：

```bash
npm run seed:demo | npx wrangler d1 execute subforge --local --file=-
```

建议的最小本地校验：

```bash
npm run test:smoke
npm run test:unit
```

## 3. Cloudflare 资源与变量

需要准备：

- 一个 D1 数据库绑定 `DB`
- 一个 KV namespace 绑定 `SUB_CACHE`
- 一个 Cron Trigger

当前仓库默认使用“无资源 ID + Worker-first”配置：所有请求先进入 Worker，API / 订阅 / 健康检查由代码处理，其余页面再回退到静态资源；这依赖 **`wrangler@4.45.0+`** 自动 provision 资源并补齐绑定。

如果你的 Cloudflare 账号环境没有自动创建出来，可在 Dashboard 中手动确认或补充以下绑定：

- `DB` → D1
- `SUB_CACHE` → KV

同时确认 `vars`：

- `APP_ENV`
- `ADMIN_LOGIN_RATE_LIMIT_WINDOW_SEC`
- `ADMIN_LOGIN_RATE_LIMIT_MAX_ATTEMPTS`
- `SUBSCRIPTION_CACHE_TTL`
- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS`
- `PREVIEW_CACHE_TTL`
- `SYNC_HTTP_TIMEOUT_MS`

### 3.1 绑定 / Secret / 调度对照

| 名称 | 类型 | 在哪配置 | 用途 | 首次部署要确认 |
| --- | --- | --- | --- | --- |
| `DB` | D1 绑定 | `wrangler.toml` + Dashboard | 存储用户、节点、模板、规则与日志 | migration 能成功执行，后台资源列表能正常读写 |
| `SUB_CACHE` | KV 绑定 | `wrangler.toml` + Dashboard | 存储预览与公开订阅缓存 | `/api/preview` 与 `/s/:token/:target` 能正常 miss / hit |
| `ASSETS` | Worker 静态资源绑定 | `[assets]` 自动生成 | 托管 `apps/web/dist` | 打开 `/` 能进入 SPA，而不是 404 或默认静态页 |
| `ADMIN_JWT_SECRET` | Runtime Secret | Dashboard 或 Wrangler Action | 管理员会话签名 | 登录后 Bearer token 能持续校验通过 |
| `0 * * * *` | Cron Trigger | `[triggers]` + Dashboard | 每小时规则源同步 | Dashboard 中能看到 Trigger 已生效 |


如果你在 Dashboard 中覆盖 `vars`，建议优先与 `wrangler.toml` 保持一致，尤其是 `SYNC_HTTP_TIMEOUT_MS`、缓存 TTL、管理员登录限流与公开订阅频控相关字段。

### 3.2 D1 备份 / 恢复 SOP

建议在以下场景前先做一次远端 D1 备份：

- production 发版前，尤其是包含 migration、手工 SQL、批量导入或人工修数
- staging 验证涉及破坏性数据操作前
- 准备做数据回滚前，先导出当前“故障现场”

建议约定：

- 备份文件统一放到 `backups/d1/` 或仓库外的受控目录，并加密保存
- 文件名至少带上环境、时间戳，最好再附 commit SHA 或 workflow run id
- SQL 导出会包含管理员密码哈希、用户 token、审计 / 同步日志等敏感数据，不要提交到 Git 仓库
- 如需异地归档，优先配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，把备份加密成 `.enc` 后再上传或外发
- 如需把 GitHub Actions 产物继续同步到对象存储 / KMS，参考第 6 节配置 `D1_BACKUP_ARCHIVE_S3_URI`、`D1_BACKUP_ARCHIVE_SSE` 与 `D1_BACKUP_ARCHIVE_KMS_KEY_ID`

常用命令：

```bash
npm run backup:d1
npm run backup:d1:staging
npm run backup:d1 -- --scope schema
npm run backup:d1 -- --scope data
D1_BACKUP_ARCHIVE_PASSPHRASE=your-passphrase npm run backup:d1:encrypted
```

如果你更想看等价的原生命令，当前脚本本质上会执行：

```bash
mkdir -p backups/d1

# production：完整导出（schema + data）
npx wrangler d1 export subforge --remote --config ./wrangler.toml --output "./backups/d1/subforge-production-$(date +%Y%m%d-%H%M%S).sql"

# staging：完整导出（schema + data）
npx wrangler d1 export subforge-staging --remote --env staging --config ./wrangler.toml --output "./backups/d1/subforge-staging-$(date +%Y%m%d-%H%M%S).sql"

# 如只想单独留 schema 或 data，可分别追加：
npx wrangler d1 export subforge --remote --config ./wrangler.toml --no-data --output "./backups/d1/subforge-production-schema-$(date +%Y%m%d-%H%M%S).sql"
npx wrangler d1 export subforge --remote --config ./wrangler.toml --no-schema --output "./backups/d1/subforge-production-data-$(date +%Y%m%d-%H%M%S).sql"
```

说明：

- `wrangler d1 export <name>` 的参数是数据库名，不是绑定名；当前仓库 production / staging 分别对应 `subforge` 与 `subforge-staging`
- 我根据本地 `wrangler@4.71.0` help 确认：默认导出的 SQL 同时包含 schema 与 data；这意味着 full export 应恢复到空库 / 新库，不能直接导回仍有原表和原数据的远端 D1
- 如需先在本地演练，把 `--remote` 换成 `--local`
- `npm run backup:d1 -- --encrypt` 或 workflow 中的加密归档会生成 `.enc` 与对应 `.sha256` 校验文件；解密时可使用 `npm run backup:d1:decrypt -- --input <backup.sql.enc>`

如果你准备让 GitHub Actions 自动把备份同步到对象存储，建议在目标 environment 中配置：

- `vars`：`D1_BACKUP_ARCHIVE_S3_URI`、`D1_BACKUP_ARCHIVE_ENDPOINT_URL`、`D1_BACKUP_ARCHIVE_AWS_REGION`、`D1_BACKUP_ARCHIVE_SSE`、`D1_BACKUP_ARCHIVE_KMS_KEY_ID`、`D1_BACKUP_ARCHIVE_STORAGE_CLASS`
- `secrets`：`D1_BACKUP_ARCHIVE_AWS_ACCESS_KEY_ID`、`D1_BACKUP_ARCHIVE_AWS_SECRET_ACCESS_KEY`、`D1_BACKUP_ARCHIVE_AWS_SESSION_TOKEN`（可选，临时凭证时使用）

workflow 实际会执行类似：

```bash
aws s3 cp backups/d1/ "s3://your-bucket/subforge-d1/<environment>/YYYY/MM/DD/gha-<run_id>/" --recursive --only-show-errors
```

约束：

- 如未配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，同步的是明文 SQL，此时必须配置 `D1_BACKUP_ARCHIVE_SSE`（例如 `AES256` 或 `aws:kms`）
- 如 `D1_BACKUP_ARCHIVE_SSE=aws:kms`，还必须配置 `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
- 如已配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，对象存储收到的是 `.enc` 与 `.sha256`
- `D1_BACKUP_ARCHIVE_ENDPOINT_URL` 可用于 Cloudflare R2、MinIO 等 S3 兼容服务；未配置时默认走 AWS 官方端点
- bucket lifecycle / prefix 留存策略需要在对象存储侧配置，不在本仓库 workflow 中声明

恢复建议：

- 先冻结发布，记录当前故障时间、commit、GitHub Actions run URL 与受影响环境
- 先把当前故障库再导出一份“事故快照”，避免二次操作覆盖现场
- 再选择恢复路径：
  - `整库恢复到空库 / 新库`：适合 full export；先在 Dashboard 准备一个空 D1 或重建同名 D1，并确保 Worker `DB` 绑定已经切到这个空库
  - `同 schema 数据回灌`：仅适合 `--no-schema` 备份，且目标库已经通过 migration 建好相同 schema
- 当前仓库没有自动化的 D1 建库 / 切库脚本，创建空库和切换绑定仍需要在 Dashboard 手工完成

常用恢复命令：

```bash
# production：把 full export 导入到已切好的空 D1 绑定
npx wrangler d1 execute DB --remote --config ./wrangler.toml --file "./backups/d1/subforge-production-YYYYMMDD-HHMMSS.sql" -y

# staging
npx wrangler d1 execute DB --remote --env staging --config ./wrangler.toml --file "./backups/d1/subforge-staging-YYYYMMDD-HHMMSS.sql" -y
```

- 如果你使用的是 `--no-schema` 数据备份，先执行 `npm run db:migrations:apply` 或 `npm run db:migrations:apply:staging`，再导入 data-only SQL
- 数据导回后，重新部署上一个稳定 Worker 版本：production 用 `npm run deploy` 或 Deploy workflow 的 `workflow_dispatch`，staging 用 `npm run deploy:staging`
- 最后至少验收：`GET /health`、`GET /api/setup/status`、管理员登录、一个公开订阅 token、用户 / 节点 / 规则源列表页；必要时再用 `npx wrangler d1 execute DB --remote --config ./wrangler.toml --command "SELECT COUNT(*) AS total FROM users;"` 抽查表数据
- 如需对下载回来的 SQL 做离线恢复演练，可执行 `npm run d1:restore:drill -- --file ./backups/d1/<backup.sql>`；若备份是 `--no-schema` data-only SQL，再追加 `--prepare-schema`
- 如需对加密备份做演练，可直接传入 `.enc`：`D1_BACKUP_ARCHIVE_PASSPHRASE=your-passphrase npm run d1:restore:drill -- --file ./backups/d1/<backup.sql.enc>`

## 4. 部署顺序

1. 在 Dashboard 或 Deploy to Cloudflare 中导入 Worker
2. 执行构建命令 `npm run build`，只生成前端静态资源
3. 执行部署命令 `npm run deploy`，由 Worker 发布阶段完成 D1 migration 与 Worker 部署
4. 若 Cloudflare 没自动补齐 D1 / KV，则在 Dashboard 手动绑定 `DB` / `SUB_CACHE` 后重试部署
5. 打开后台，完成首次安装向导创建管理员
6. 如有需要再导入 demo 数据
7. 如需单独检查前端构建产物，可再执行一次 `npm run build:web`

如需通过 CLI 收敛为单条命令，可在完成 Cloudflare Secrets 与绑定配置后执行：

```bash
npm run init:remote -- --admin-user admin --admin-password your-password
```

这条命令会串行执行 `npm run test:smoke`、`npm run build`、`npm run deploy`，并把首个管理员直接写入远端 D1。

常用命令：

```bash
npm run init:local
npm run init:remote -- --admin-user admin --admin-password your-password
npm run build
npm run deploy
npm run db:migrations:apply
npm run build:web
```

如果页面出现 `Unexpected token '<'` 之类 JSON 解析错误，先确认当前部署已使用这版 `run_worker_first = true` 配置，再继续排查绑定问题；详细排查请看 `docs/排障与常见问题.md`。

## 5. 部署后最小验收

建议至少验证以下 5 点：

1. `GET /health` 返回 `200`
2. `GET /api/setup/status` 返回 JSON
3. 后台页面能正常打开，并可完成首次安装向导或管理员登录
4. 任意一个公开订阅 token 能返回订阅原文
5. 如果你已导入规则源，最近同步日志里的 `details_json` 应能看到格式、阶段、错误码、耗时等结构化信息

如果需要更完整的巡检、告警和事件响应流程，请看 `docs/运维Runbook与告警处理.md`。

### 5.1 推荐的首轮验收顺序

建议按下面顺序完成首轮验收：

1. 打开 `/`，确认能进入后台 SPA，而不是 Worker 默认 404 页面
2. 请求 `/health` 与 `/api/setup/status`，确认健康检查和 JSON API 都正常
3. 如实例尚未初始化，在首次安装向导创建管理员；如已 seed，则直接登录后台
4. 如果已导入 demo 数据或已有真实用户，验证一个公开订阅地址，例如 `/s/:token/mihomo` 或 `/s/:token/singbox`
5. 手动同步一个规则源，确认同步详情中能看到 `stage`、`errorCode`、`operatorHint`、`retryable`；失败时还能看到 `supportedShapes` 与 `contentPreview`
6. 在节点页执行一次节点 JSON 批量导入，确认节点数量、审计日志与后续预览数据都会更新
7. 如启用了远程节点源，再执行一次“远程节点源同步”，确认节点数、同步日志与审计日志都会更新
8. 手动点击“重建订阅缓存”，再重新访问预览或公开订阅，确认最新内容能重新生成

### 5.2 部署后首轮排障观察点

- `/api/*` 返回 HTML 或前端出现 `Unexpected token '<'`：优先检查 `wrangler.toml` 中 `[assets]`、`directory = "./apps/web/dist"` 与 `run_worker_first = true`，并确认部署前执行过 `npm run build`
- 后台空白、首次安装向导异常或登录失败：检查 `ADMIN_JWT_SECRET` 是否已配置，以及 D1 `DB` 是否绑定到当前 Worker
- 规则源长期停留在 `failed`：先看“同步状态 / 同步日志”，重点观察 `details_json` 中的 `stage`、`errorCode`、`operatorHint`、`retryable`、`supportedShapes` 与 `contentPreview`
- 远程节点源同步失败：先确认上游是否返回 JSON 数组或 `{"nodes":[...]}`，再看后台同步日志与审计日志
- Deploy 成功但数据未生效：确认远端 migration 是否真的执行过，可重新执行 `npm run db:migrations:apply`
- Cron 没有按预期触发：确认 Dashboard 中 Trigger 已存在，且当前 Trigger 绑定的是同一个 Worker 服务

## 6. GitHub Actions 自动化

当前仓库已提供四个自动化 workflow：

- `.github/workflows/ci.yml`：在 push / pull request / `workflow_dispatch` 时执行 `npm run ci:verify`，依次覆盖 `npm run test:contract`、`npm run test:smoke`、`npm test`、`npm run test:unit`、`npm run typecheck`、`npm run build`、`npm run build:worker`
- `.github/workflows/deploy.yml`：`release/* -> staging`、`main -> production`；也支持 `workflow_dispatch`，可显式指定 `target_environment` 与 `git_ref`。如果手动选择 `production`，必须从 `main` 分支上下文触发
- `.github/workflows/d1-backup.yml`：每天 `03:15 UTC` 定时为 production 执行 full D1 导出；手动触发时也可选择 `staging` / `production`、`full|schema|data` 与 artifact 保留天数；如配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，会上传加密后的 `.enc` 与 `.sha256`
- `.github/workflows/failure-alert.yml`：当 `CI`、`Deploy` 或 `D1 Backup` 失败时生成 failure summary；如配置 `FAILURE_WEBHOOK_URL`，还会把失败信息推送到外部 webhook

部署 / 归档相关的 GitHub Secrets 与 vars：

- `CLOUDFLARE_API_TOKEN`
- `CLOUDFLARE_ACCOUNT_ID`
- `ADMIN_JWT_SECRET`
- `FAILURE_WEBHOOK_URL`
- `D1_BACKUP_ARCHIVE_PASSPHRASE`
- `D1_BACKUP_ARCHIVE_S3_URI`
- `D1_BACKUP_ARCHIVE_ENDPOINT_URL`
- `D1_BACKUP_ARCHIVE_AWS_REGION`
- `D1_BACKUP_ARCHIVE_SSE`
- `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
- `D1_BACKUP_ARCHIVE_STORAGE_CLASS`
- `D1_BACKUP_ARCHIVE_AWS_ACCESS_KEY_ID`

补充说明：

- workflow 当前默认使用 `npm ci`，并依赖已提交的 `package-lock.json`
- 当前 workflow 会先执行对应环境的远端 migration，再通过 `cloudflare/wrangler-action@v3` 在 `apps/worker` 目录下执行对应 `wrangler deploy` 命令，并上传 `ADMIN_JWT_SECRET` 到 Worker Secret
- 如果 `workflow_dispatch` 选择 `production`，但当前运行上下文不是 `main`，workflow 会显式失败，避免从错误分支误发 production
- 如果配置了 `D1_BACKUP_ARCHIVE_S3_URI`，workflow 会继续执行类似 `aws s3 cp backups/d1/ "s3://.../<environment>/YYYY/MM/DD/gha-<run_id>/" --recursive --only-show-errors`
- 明文 SQL 归档时必须配置 `D1_BACKUP_ARCHIVE_SSE`；如使用 `aws:kms`，还要补 `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
- `D1_BACKUP_ARCHIVE_ENDPOINT_URL` 可用于 Cloudflare R2 / MinIO；bucket lifecycle / prefix 留存策略需要在对象存储侧配置
- 当前仓库没有自动 down migration；如 schema 已损坏，仍需要依赖 Cloudflare Dashboard 和备份恢复完成回退

本地镜像 workflow 行为时，常用命令是：

```bash
npm run ci:verify
npm run build:worker:staging
npm run d1:restore:drill -- --file ./backups/d1/<backup.sql>
```

如在受限沙箱或只读 home 目录里执行本地 dry-run，Wrangler 可能无法写入默认日志目录；此时可临时使用：

```bash
XDG_CONFIG_HOME=/tmp WRANGLER_OUTPUT_FILE_DIRECTORY=/tmp npm run ci:verify
```
