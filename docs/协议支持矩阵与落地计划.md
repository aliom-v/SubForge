# SubForge 协议支持矩阵与落地计划

## 1. 文档目的

本文档用于统一说明 SubForge 当前对各协议的真实支持层级，以及后续建议的落地顺序。

这里的重点不是“协议名有没有出现在页面上”，而是要拆清以下五个层级：

1. 后台手动录入
2. 分享链接导入
3. 订阅 URL 远程预览导入
4. 结构化协议向导
5. 订阅输出链路

如果不把这五层拆开，很容易出现“看起来支持，但实际上只支持其中一半”的误解。

---

## 2. 如何理解“支持”

### 2.1 后台手动录入

表示管理员可以在“节点”页直接创建该协议节点，自己填写：

- `name`
- `protocol`
- `server`
- `port`
- `credentials`
- `params`

这层是最底线能力。

### 2.2 分享链接导入

表示 `packages/core` 中的共享解析器已经能把对应分享链接转换为统一节点结构，再交给后台批量创建。

### 2.3 订阅 URL 远程预览导入

表示 Worker 可以先抓取远程订阅文本，再复用同一份解析器做一次性预览和批量创建。

### 2.4 结构化协议向导

表示 Web 节点页里已经有协议专用字段向导，管理员不需要直接手写全部 JSON。

### 2.5 订阅输出链路

表示节点可以进入当前 `mihomo` / `singbox` 输出链路。

注意：

- 这一层是“统一节点模型可进入渲染器”
- 不是“平台已经对该协议做完全部协议级产品化适配”

当前渲染器是通用透传思路：

- `mihomo` 输出会写入 `type/server/port` 并展开 `credentials` / `params`
- `singbox` 输出会写入 `type/server/server_port` 并展开 `credentials` / `params`

因此，某协议即使能输出，也仍然依赖管理员自己把 metadata 填对。

## 3. 当前协议支持矩阵

| 协议 / 能力 | 手动录入 | 分享链接导入 | 远程预览导入 | 结构化向导 | 订阅输出 | 当前结论 |
| --- | --- | --- | --- | --- | --- | --- |
| `vless` | 支持 | 支持 | 支持 | 支持 | 支持 | 当前最完整 |
| `trojan` | 支持 | 支持 | 支持 | 支持 | 支持 | 当前最完整 |
| `vmess` | 支持 | 支持 | 支持 | 支持 | 支持 | 当前最完整 |
| `ss` | 支持 | 支持 | 支持 | 支持 | 支持 | 五层已打通，向导当前聚焦常见字段 |
| `ssr` | 支持 | 支持 | 支持 | 支持 | 支持 | 已进入可用状态，复杂变体仍建议核对 JSON |
| `tuic` | 支持 | 支持 | 支持 | 支持 | 支持 | 已进入可用状态，复杂变体仍建议核对 JSON |
| `hysteria2` | 支持 | 支持 | 支持 | 支持 | 支持 | 常见字段已打通，复杂变体仍未补 |
| 其他自定义协议 | 支持 | 未支持 | 未支持 | 未支持 | 支持 | 依赖手动 JSON，自担字段正确性 |

## 4. 当前真实边界

### 4.1 已经落地的“常见可导入协议”

当前已落地：

- `vless://`
- `trojan://`
- `vmess://`
- `ss://`
- `ssr://`
- `tuic://`
- `hysteria2://` / `hy2://`

并且支持：

- 粘贴多行分享链接
- 粘贴整段 Base64 包装订阅文本
- 输入订阅 URL 做一次性远程抓取预览

### 4.2 已经落地的“结构化协议向导”

当前已支持：

- `vless`
- `trojan`
- `vmess`
- `ss`
- `ssr`
- `tuic`
- `hysteria2`

这意味着：

- 这七类协议都可以通过结构化字段回填 `credentials` / `params`
- `ss` / `hysteria2` 当前仍只覆盖常见主路径字段
- 其余协议继续直接编辑原始 JSON

### 4.3 当前尚未落地的能力

当前仍未落地：

- `hysteria2` 多端口 `authority` 与更多复杂变体
- 完整的 Clash / Mihomo / sing-box 配置编辑器
- 远程节点源持久化与定时同步

### 4.4 一个重要区分

当前“能输出”不等于“完整支持”。

例如：

- `hysteria2` 现在可以手动录入后进入输出链路
- 现在也已经有分享链接导入与远程预览
- 现在也已经有常见字段结构化向导
- 现在也已经有专门的协议字段校验

但它依然不等于“所有变体都支持”，尤其多端口与更复杂组合仍未补齐。

### 4.5 当前已知细粒度限制

为了避免“文档里写支持，实际一上手还是踩坑”，当前还需要记住：

1. `ss` 虽然已经进入结构化向导，但当前向导只覆盖 `cipher`、`password`、`plugin`
2. `ssr` / `tuic` 已进入导入、向导与输出链路，但仍不等于“所有变体完全产品化”
3. `ss` 的更复杂插件参数、特殊变体与非主路径字段，仍建议继续直接核对 JSON
4. `hysteria2` 目前已支持 `hysteria2://` / `hy2://` 常见单端点分享链接导入
5. `hysteria2` 当前已支持常见字段结构化向导与协议级字段校验，但仍不是“复杂变体全覆盖”
6. `hysteria2` authority 里的多端口写法当前还不支持，遇到这类链接时仍需要手动 JSON 录入
7. `hysteria2` 未知 query 参数当前会直接报 unsupported，而不是静默忽略
8. 完整配置导入当前已经进入“节点 + 模板骨架 + 模板结构助手”模式，但仍不是完整配置编辑器
9. 链式代理当前通过 `params.upstreamProxy` 统一表达，已支持 group 级展开与模板页可视化配置，但仍不是独立图模型

## 5. 推荐的下一阶段顺序

### 第一阶段：继续补复杂变体和完整配置编辑深度

建议优先级：

1. `hysteria2` 多端口 `authority` 与更多复杂分享链接变体
2. 模板结构助手的下一层字段级校验与去重提示
3. 更强的链式代理图形化编辑
当前已支持节点页 group 级链路展开和模板页 `proxy-group` 可视化配置，下一步目标是更完整的拖拽式图编辑
4. 远程节点源持久化与定时同步

原因：

- 第二十二批已经把 `hysteria2` 向导、字段校验、示例库、排障文档和长链路回归补到位
- 当前更大的缺口已经从“常见主路径是否可用”转成“复杂变体和更大输入范围是否可用”
- 继续补协议前，现有 `ss` / `hysteria2` 基线已经足够稳定，可以转向下一阶段能力扩展

上一阶段的实际落地结果已单独整理到：

- `docs/archive/自动化验证执行记录/第二十二批自动化验证执行清单.md`

上一阶段完成标准：

- 协议既能导入，也能在后台用结构化字段编辑
- 对应文档、示例参考、排障清单和回归测试同步齐全

### 第二阶段：扩展下一批主流协议与复杂字段覆盖

建议候选：

- 更多 Reality 相关分享链接细节补齐
- 其他仍具用户需求的单链接协议

完成标准：

- 至少补齐导入解析
- 若进入常见协议列表，最好同步补结构化向导

### 第三阶段：做“完整配置产品化”而不是只做配置导入

建议范围：

- Mihomo `proxy-groups` 的更深字段级建模与 provider 关系编辑
- sing-box `route.rules` / static outbounds 的更深字段级建模
- 更强的字段校验、重复导入提示与冲突处理
- 节点去重、重复导入识别、冲突处理

完成标准：

- 不再只依赖管理员手工核对 JSON
- 导入体验从“能导”升级到“可运营”

### 第四阶段：补远程节点源能力

建议范围：

- 远程节点源持久化建模
- 定时同步
- 变更对比与审计
- 节点源级缓存失效与状态展示

完成标准：

- 远程 URL 不只是一次性预览
- 后台可以长期管理远程节点来源

## 6. 后续补协议时的落地清单

这份文档不只是给当前用户看边界，也可以直接作为后续补协议时的执行清单。

### 6.1 代码落地点

如果要把一个协议从“手动 JSON 可填”推进到“更完整支持”，至少应检查这些位置：

1. `packages/core/src/node-import.ts`
2. `apps/web/src/node-metadata.ts`
3. `apps/web/src/App.tsx`
4. `apps/worker/src/index.ts`
5. `tests/web-node-import.test.mjs`
6. `tests/web-node-metadata.test.mjs`
7. `tests/worker-write-integration.test.mjs`
8. `scripts/smoke-check.mjs`

对应含义：

- `packages/core/src/node-import.ts`：补分享链接解析、错误提示、Base64 文本解包后的兼容性
- `apps/web/src/node-metadata.ts`：如果协议进入常见路径，补结构化字段向导、示例 JSON 与回填逻辑
- `apps/web/src/App.tsx`：同步节点页说明、placeholder、按钮提示与支持协议文案
- `apps/worker/src/index.ts`：如果有新的预览或写入语义变化，同步接口校验与错误语义
- 测试与 smoke：把导入、向导、请求语义、文档入口一起钉住

### 6.2 文档与状态同步点

每补一个协议，至少同步更新：

1. `README.md`
2. `docs/节点管理与订阅使用说明.md`
3. `docs/部署指南.md`
4. `docs/实施方案.md`
5. `docs/已知问题与修复计划.md`
6. `docs/自动化验证与CI计划.md`
7. `.omx/plans/2026-03-07-subforge-roadmap.md`

### 6.3 文档写法约束

补协议时，文档里要明确写清它到底补到了哪一层：

1. 只补手动录入，就写“手动 JSON 录入”
2. 只补分享链接导入，就写“导入已支持，向导未补”
3. 只有进入 `mihomo` / `singbox` 输出，不要写成“完整协议支持”
4. 只有“导入 + 远程预览 + 结构化向导 + 输出”基本打通后，才适合写成当前主路径

### 6.4 建议验收口径

一个协议要从“新增支持”进入“可稳定使用”，建议至少满足：

1. 分享链接导入可解析
2. 订阅 URL 远程预览可复用同一解析结果
3. 页面说明和按钮提示已经同步
4. 如果进入常见协议列表，结构化向导已补齐
5. smoke 与对应回归测试已经覆盖

## 7. 当前建议用法

如果你现在要稳定使用，建议按下面理解：

1. `vless` / `trojan` / `vmess`：可优先作为当前主路径
2. `ss`：当前也可作为常见路径使用，但创建前后仍建议核对 JSON
3. `hysteria2`：当前可通过分享链接或远程预览导入，但编辑仍建议熟悉协议字段的人继续手动核对 JSON
4. 其他协议：默认按自定义节点处理，不要把“能填 protocol 名”误当成“平台已经完整支持”
