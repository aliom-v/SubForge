# 导入成功但订阅没变化排障指南

## 1. 适用场景

适用于下面这类现象：

- “分享链接已经解析成功，也创建出节点了”
- “节点列表里能看到新节点”
- “但预览页或真实订阅看起来没变化”

这类问题通常不是“导入失败”，而是后半段链路断了。

---

## 2. 推荐排查顺序

### 2.1 先看“节点”页导入反馈

先确认你看到的是哪一种成功：

1. 只是出现了解析结果表格
2. 还是已经点击了 `批量创建 N 个节点`
3. 还是已经把单条结果载入“创建节点”表单并真正提交保存

如果只是“解析成功”，但没有真正创建节点，后面的用户绑定和订阅当然不会变化。

页面上至少应该看到：

- “有效行 N”
- 节点结果表格
- 或“节点已创建，请到‘用户’页完成绑定并在‘预览’页验证输出”

### 2.2 再看“用户”页是否真的完成绑定

这是最常见断点。

必须确认：

1. 目标用户是否正确
2. 新节点是否已经勾选
3. 保存绑定后是否仍保持勾选状态

如果用户没有绑定任何启用节点，预览和真实订阅都不会包含新节点。

### 2.3 检查“模板”页目标类型

至少确认：

1. 对应目标类型存在启用中的默认模板
2. 你预览的是 `mihomo` 还是 `singbox`
3. 真实订阅地址与模板目标类型一致

常见误判：

- 只创建了 `mihomo` 默认模板，却去拉 `singbox`
- 模板存在，但不是默认模板
- 模板被禁用了

### 2.4 到“预览”页确认后台链路

推荐先用后台预览定位，不要直接跳到公开订阅。

操作顺序：

1. 选择目标用户
2. 选择目标类型
3. 点击刷新预览

这里应该能直接回答三个问题：

- 编译有没有报错
- 当前输出里有没有新节点名称
- 模板和规则是否按预期生效

如果预览页已经能看到新节点，而真实订阅没变化，再去查公开订阅路径。

### 2.5 最后看真实订阅地址

公开订阅地址格式：

- `/s/:token/mihomo`
- `/s/:token/singbox`

需要确认：

1. 用的是目标用户当前有效 token
2. 用户状态仍是启用
3. 用户没有过期
4. 请求的目标类型与默认模板匹配

### 2.6 “预览有变化但公开订阅没变化”怎么查

如果后台预览已经能看到新节点，但公开订阅看起来还是旧内容，优先按这个分支排查：

1. 预览和公开订阅是不是同一个目标类型
2. 公开订阅地址里的 `token` 是不是当前用户最新 token
3. 你访问的是否还是旧的订阅地址，尤其是重置 token 之后
4. 客户端或浏览器是否缓存了旧内容

推荐顺序：

1. 先在后台“预览”页确认最新内容确实已经生成
2. 直接在浏览器打开 `/s/:token/mihomo` 或 `/s/:token/singbox` 复核
3. 如果浏览器里已经是新内容，但客户端还是旧内容，优先怀疑客户端缓存或错误订阅地址
4. 如果浏览器里也是旧内容，再回头检查 token、目标类型和默认模板

常见误判：

- 预览看的是 `mihomo`，公开订阅拉的是 `singbox`
- 用户重置过 token，但客户端里还是旧地址
- 订阅客户端自身有缓存，没有真正重新拉取
- 看的不是目标用户对应的公开订阅地址

---

## 3. 针对 `ss` / `hysteria2` 的额外检查

### 3.1 `ss`

当前主路径要求至少可核对：

- `credentials.cipher`
- `credentials.password`
- `params.plugin`（如果你用了插件）

如果你写的是更复杂的 plugin 字段，继续建议直接核对原始 JSON。

### 3.2 `hysteria2`

当前主路径优先检查：

- `credentials.password`
- `params.sni`
- `params.obfs`
- `params["obfs-password"]`
- `params.alpn`
- `params.insecure`

当前已知限制：

- 多端口 `authority` 仍不支持
- 未知 query 参数会直接报 unsupported
- 更复杂组合仍建议手动核对 JSON

---

## 4. 一条最小排障路径

如果你只想最快定位问题，按这个顺序：

1. “节点”页确认已经真正创建成功
2. “用户”页确认绑定已保存
3. “模板”页确认默认模板启用且目标类型正确
4. “预览”页确认内容里是否出现新节点
5. 浏览器访问真实订阅地址复核

只要前四步里有一步没过，通常就不是公开订阅本身的问题。
