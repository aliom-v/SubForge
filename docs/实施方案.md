# SubForge 实施方案

## 1. 文档目的

本文档用于把 `项目介绍.md` 中的产品设想收敛为一份可以直接执行的实施方案，目标是：

- 明确第一阶段做什么、不做什么
- 明确仓库结构、模块边界和职责
- 明确 D1 / KV / Worker / Web 的落地方式
- 明确 API、数据模型、缓存和同步流程
- 明确分阶段里程碑，保证后续可以逐步实现

本文档默认面向单仓库开发，优先保证：

- 能部署
- 能维护
- 能从 MVP 平滑演进

说明：

- 本文档描述的是目标架构与目标落地路径，不等同于“当前仓库已经全部完成”
- 当前真实实现范围、已确认偏差与修复顺序，以 `docs/已知问题与修复计划.md` 为准

## 2. 项目定义

### 2.1 项目定位

SubForge 是一个部署在 Cloudflare 上的订阅管理平台，负责统一管理：

- 节点
- 用户
- 订阅 token
- 模板
- 规则源
- 同步任务

并输出：

- Mihomo 订阅
- sing-box 订阅

### 2.2 核心原则

SubForge 必须坚持以下原则：

1. 不运行代理核心，只做配置管理与订阅分发
2. 不依赖常驻进程，优先使用 Cloudflare Worker + Cron
3. 不做动态脚本引擎，所有转换逻辑都显式实现
4. 先做模板化、配置化能力，再做高级扩展
5. 优先保证订阅稳定输出，再优化后台体验

### 2.3 第一阶段目标

当前阶段建议把 MVP 收敛为以下能力：

- 管理员登录
- 节点 CRUD
- 用户管理（创建 / 编辑 / 删除 / 启停 / token 管理 / 节点绑定）
- 模板管理（创建 / 更新 / 删除 / 默认模板切换）
- 规则源管理（创建 / 更新 / 删除 / 启停）
- 手动 / Cron 同步规则源
- 生成 Mihomo 订阅
- 生成 sing-box 订阅
- 基于 KV 的订阅缓存
- 基础操作日志与同步日志

### 2.4 非目标

以下能力不在 MVP 范围内：

- 代理测速
- 代理常驻运行
- 用户自定义 JS 执行
- 多租户计费系统
- 复杂权限系统
- 超大规模实时批处理

## 3. 技术选型

### 3.1 总体选型

- 前端：React + Vite + TypeScript
- API：Cloudflare Workers + TypeScript
- 数据库：Cloudflare D1
- 缓存：Cloudflare KV
- 定时任务：Cloudflare Cron Triggers
- 部署：Wrangler
- 代码组织：Monorepo
- 包管理：`npm workspaces`

### 3.2 选型理由

选择 `npm workspaces` 而不是额外引入复杂 monorepo 工具，原因是：

- 当前项目规模不大，先降低初始化成本
- Node / Vite / Wrangler 对 npm 支持最直接
- 后续如果包数量变多，再迁移到更强的 workspace 管理工具也不迟

### 3.3 可选增强

以下能力作为后续增强项保留：

- R2：导入文件、规则快照备份
- Cloudflare Access：保护后台入口
- GitHub Actions：自动部署与检查
- Sentry / Logpush：错误监控

## 4. 仓库结构设计

建议采用如下目录：

```text
.
├── apps/
│   ├── web/                # 管理后台
│   └── worker/             # Worker API
├── packages/
│   ├── core/               # 订阅生成、同步、缓存失效等核心逻辑
│   └── shared/             # 类型、常量、schema、通用工具
├── migrations/             # D1 migration SQL
├── scripts/                # 本地初始化与辅助脚本
├── docs/                   # 项目文档
├── package.json            # workspace 根配置
├── wrangler.toml           # Worker 部署配置
└── README.md
```

### 4.1 目录职责

- `apps/web`：后台页面、表单、列表、配置预览、同步入口
- `apps/worker`：HTTP API、鉴权、订阅输出、Cron 任务
- `packages/core`：业务服务层，不直接依赖 UI
- `packages/shared`：共享 DTO、zod schema、常量枚举、工具函数
- `migrations`：数据库结构演进
- `scripts`：初始化 demo 数据、导入样例、开发辅助脚本
- `docs`：实施文档、接口约定、里程碑计划

## 5. 系统架构

### 5.1 架构概览

系统由四个核心层组成：

1. Web 管理后台
2. Worker API
3. D1 数据存储
4. KV 缓存层

外加一个定时调度层：

5. Cron 任务

### 5.2 请求流

#### 管理后台请求流

1. 管理员登录后台
2. 前端调用 Worker API
3. Worker 完成参数校验和鉴权
4. Worker 写入或查询 D1
5. 变更成功后触发缓存失效
6. 前端刷新列表或预览结果

#### 订阅输出请求流

1. 客户端访问 `/s/:token/mihomo` 或 `/s/:token/singbox`
2. Worker 校验 token 和用户状态
3. 优先读取 KV 缓存
4. 缓存命中则直接返回
5. 缓存未命中则读取 D1 所需配置
6. 使用 `packages/core` 生成最终订阅内容
7. 写入 KV 后返回客户端

#### 定时同步请求流

1. Cron 触发同步任务
2. Worker 拉取远程规则源
3. 解析后写入 D1
4. 记录同步日志
5. 清理受影响用户的订阅缓存

## 6. 模块设计

### 6.1 管理员认证模块

当前实现使用最简单且可维护的方案：

- 管理员账号存 D1
- 密码存 hash，不存明文
- 登录成功后签发会话 token
- token 通过 Bearer token 传递

当前仓库行为：

- API 侧支持 Bearer token
- Web 侧使用 `localStorage` 保存 token
- `/api/admin/logout` 会同时清理前端本地登录态，并在服务端写入管理员会话撤销时间

后续如要增强，可引入：

- HTTP-only cookie session
- Cloudflare Access
- 单点登录
- 多角色权限模型

### 6.2 用户与订阅模块

每个用户至少拥有：

- 唯一 token
- 状态（启用 / 禁用）
- 到期时间
- 可用节点集合
- 可用模板集合

订阅生成时必须校验：

- token 是否存在
- 用户是否启用
- 是否已过期
- 是否存在可用节点
- 是否存在目标格式模板

### 6.3 节点管理模块

当前实现支持手动录入节点、常见分享链接导入、Base64 包装订阅文本自动解包，以及订阅 URL 的一次性远程抓取预览导入；当前已覆盖 `vless` / `trojan` / `vmess` / `ss` / `hysteria2` 导入，其中 `ss` 已补结构化向导，`hysteria2` 仍主要依赖导入后核对 JSON，远程节点源同步保留为后续扩展。

协议支持需要拆成不同层级理解：

- 手动录入支持
- 分享链接导入支持
- 远程预览导入支持
- 结构化协议向导支持
- 订阅输出支持

当前不要把“能进入统一渲染器输出”直接等同于“该协议已经完整产品化支持”。

详细矩阵与后续落地顺序统一见：

- `docs/协议支持矩阵与落地计划.md`

当前节点写接口约束：

- `sourceType` 仅接受 `manual`
- `sourceId` 保留给后续节点源同步能力，当前写接口会拒绝非空值
- `credentials` / `params` 仅接受 JSON object；更新时可传 `null` 显式清空
- Web 侧当前提供基础字段与 `credentials` / `params` JSON 手动录入，并支持 `vless://` / `trojan://` / `vmess://` / `ss://` / `hysteria2://` 分享链接导入、Base64 包装订阅文本自动解包，以及订阅 URL 的一次性远程预览导入；`vless` / `trojan` / `vmess` / `ss` / `hysteria2` 已提供常见字段的结构化向导
- `ss` 结构化向导当前主要覆盖 `cipher` / `password` / `plugin` 三类常见字段
- `hysteria2` 当前已支持 `hysteria2://` / `hy2://` 常见单端点分享链接导入、常见字段结构化向导与协议级字段校验，但多端口 authority 还未覆盖
- 若导入后节点已经存在，但订阅输出仍未变化，实际排查顺序仍应回到“节点是否创建成功 -> 是否已绑定用户 -> 是否存在启用中的默认模板”

节点必须统一存储为规范化结构，而不是把各种原始文本直接散落保存。推荐字段包括：

- 节点名称
- 协议类型
- 服务地址
- 端口
- 凭据 / UUID / 密钥等配置
- 附加参数 JSON
- 来源类型
- 启用状态
- 最后同步时间

### 6.4 模板管理模块

模板用于定义最终配置的骨架，不允许在运行期执行任意脚本。模板模型建议：

- `target_type`：`mihomo` 或 `singbox`
- `content`：模板正文
- `version`：版本号
- `is_default`：是否默认模板
- `enabled`：是否启用

当前模板接口约束：

- 只有 `enabled` 模板才能被创建为默认模板
- 默认模板在被禁用时会自动清除默认标记
- `set-default` 只接受当前可生效的模板
- 避免出现“默认模板标记已写入，但订阅选择逻辑不会真正采用它”的状态分裂

模板渲染由 `packages/core` 负责，Web 只负责编辑与预览。

### 6.5 规则源模块

规则源必须显式管理，至少包括：

- 名称
- URL
- 格式类型
- 是否启用
- 最后同步时间
- 失败次数

同步时需要：

- 拉取远程内容
- 校验响应合法性
- 记录 hash，避免重复写入
- 写同步日志
- 触发缓存失效

### 6.6 缓存模块

缓存对象至少包括：

- `sub:mihomo:{token}`
- `sub:singbox:{token}`
- `template:default:{target}`
- `rules:snapshot:{sourceId}`

缓存策略：

- 用户、节点、模板、规则变更后主动失效
- 订阅请求时采用读缓存优先
- 对订阅结果设置 TTL，避免长期脏数据

MVP 推荐订阅缓存 TTL：

- 15 分钟到 60 分钟之间

建议默认值：

- 30 分钟

### 6.7 同步与任务模块

当前实现中，Cron 负责：

- 同步启用的远程规则源
- 记录同步任务结果
- 按规则源变化失效相关订阅缓存

MVP 先支持：

- 手动同步
- 固定频率的自动同步

不做：

- 复杂任务编排
- 多阶段任务队列

## 8. API 设计

API 分为三类：

- 健康检查接口
- 后台管理接口
- 公开订阅接口

### 8.1 健康检查

#### `GET /health`

用途：服务健康检查

返回建议：

```json
{
  "ok": true,
  "service": "subforge-worker",
  "time": "2026-03-07T00:00:00.000Z"
}
```

### 8.2 管理员认证接口

#### `POST /api/admin/login`

用途：管理员登录

请求示例：

```json
{
  "username": "admin",
  "password": "******"
}
```

成功返回：

```json
{
  "ok": true,
  "token": "session-token",
  "admin": {
    "id": "adm_001",
    "username": "admin",
    "role": "admin"
  }
}
```

#### `GET /api/admin/me`

用途：获取当前登录管理员信息

#### `POST /api/admin/logout`

用途：退出登录

当前语义：

- 前端会清理本地保存的 token
- 服务端会把当前管理员的 `session_not_before` 更新为最新时间
- 受保护接口会拒绝 `iat` 早于撤销时间的旧 token，并返回 `401`

成功返回示例：

```json
{
  "ok": true,
  "data": {
    "loggedOut": true,
    "serverRevocation": true,
    "mode": "server_revoked",
    "revokedAt": "2026-03-08T00:00:00.000Z"
  }
}
```

### 8.3 用户管理接口

#### `GET /api/users`

用途：用户列表

#### `POST /api/users`

用途：创建用户

建议字段：

- `name`
- `expires_at`
- `remark`

#### `PATCH /api/users/:id`

用途：更新用户

#### `POST /api/users/:id/reset-token`

用途：重置 token

#### `POST /api/users/:id/nodes`

用途：绑定用户可用节点

### 8.4 节点管理接口

#### `GET /api/nodes`

用途：节点列表

#### `POST /api/nodes`

用途：创建节点

#### `PATCH /api/nodes/:id`

用途：更新节点

说明：Web 当前支持通过 JSON 文本录入 `credentials` / `params`，用于补充协议凭据与扩展参数。

#### `DELETE /api/nodes/:id`

用途：删除节点

### 8.5 模板管理接口

#### `GET /api/templates`

#### `POST /api/templates`

约束：若 `isDefault=true`，则模板必须同时处于启用状态。

#### `PATCH /api/templates/:id`

约束：disabled 模板不能被更新为默认模板，除非同一请求中先恢复为启用状态。
约束：若当前默认模板被禁用，其默认标记会在同一次更新中自动清空。

#### `POST /api/templates/:id/set-default`

约束：仅允许把 enabled 模板设为默认模板。

### 8.6 规则源接口

#### `GET /api/rule-sources`

#### `POST /api/rule-sources`

#### `PATCH /api/rule-sources/:id`

#### `POST /api/rule-sources/:id/sync`

用途：手动同步规则源

### 8.7 订阅预览接口

#### `GET /api/preview/:userId/mihomo`

#### `GET /api/preview/:userId/singbox`

用途：后台预览生成结果，不走公开 token 路径

### 8.8 公开订阅接口

#### `GET /s/:token/mihomo`

返回：`text/plain` 或 YAML

#### `GET /s/:token/singbox`

返回：`application/json`

### 8.9 接口错误格式

建议统一返回：

```json
{
  "ok": false,
  "error": {
    "code": "USER_DISABLED",
    "message": "user is disabled"
  }
}
```

## 9. 订阅生成设计

### 9.1 生成流程

订阅生成服务需要执行以下步骤：

1. 根据 token 查找用户
2. 校验用户状态和到期时间
3. 查询用户绑定的可用节点
4. 读取目标格式模板
5. 读取启用规则快照
6. 组装标准中间数据模型
7. 渲染为目标格式文本
8. 写入缓存并返回

### 9.2 中间模型

为了避免不同输出格式直接耦合数据库结构，`packages/core` 需要定义统一中间模型，例如：

- `SubscriptionUser`
- `SubscriptionNode`
- `SubscriptionRuleSet`
- `SubscriptionTemplate`
- `CompiledSubscription`

### 9.3 渲染器设计

推荐按目标格式拆成两个 renderer：

- `renderMihomoSubscription()`
- `renderSingboxSubscription()`

两者共享统一输入模型，但输出格式独立，避免后续维护混乱。

### 9.4 模板策略

模板必须是“有限变量替换 + 明确结构插槽”，不允许执行任意脚本。支持的插槽示例：

- `{{proxies}}`
- `{{proxy_groups}}`
- `{{rules}}`
- `{{dns}}`
- `{{outbounds}}`

## 10. 同步设计

### 10.1 同步对象

MVP 先支持：

- 规则源同步
- 节点源同步留作后续扩展

### 10.2 同步流程

1. 读取启用的 source 配置
2. 请求远程 URL
3. 校验响应状态码与内容类型
4. 解析原始内容
5. 转换为内部标准结构
6. 比较 hash 判断是否有变化
7. 写入 D1
8. 写入 `sync_logs`
9. 失效相关缓存

### 10.3 失败处理

同步失败时必须：

- 记录失败日志
- 增加失败计数
- 保留上一个可用快照
- 不用失败结果覆盖已有有效数据

---

## 11. 缓存与失效策略

### 11.1 缓存 Key 约定

建议使用以下 key：

- `sub:mihomo:{token}`
- `sub:singbox:{token}`
- `preview:mihomo:{userId}`
- `preview:singbox:{userId}`
- `template:default:{targetType}`
- `rules:active:{targetType}`

### 11.2 失效触发点

以下操作必须触发缓存失效：

- 用户状态变化
- 用户 token 重置
- 用户绑定节点变化
- 节点内容变化
- 默认模板变化
- 规则快照更新

### 11.3 缓存原则

- 公开订阅接口可缓存
- 后台预览接口可短缓存或不缓存
- 缓存失效优先保证正确性，其次才是命中率

---

## 12. 安全设计

### 12.1 基本安全要求

MVP 需要满足：

- 密码 hash 存储
- 管理接口必须鉴权
- 输入参数必须校验
- 敏感信息不得写入前端明文
- 日志中不得记录完整密码与私钥

### 12.2 Token 安全

- 用户 token 使用高熵随机字符串
- 支持重置 token
- token 泄露后可立即失效旧 token

### 12.3 风险边界

MVP 不实现复杂 RBAC，但需要预留 `role` 字段；
后续如果扩展多管理员，再增加细粒度权限。

## 15. 实施优先级

建议按以下顺序实现：

1. 先把工程初始化完成
2. 再完成 `shared` 与 `core`
3. 再完成 D1 migration 和 Worker API
4. 再做 Web 管理后台
5. 最后补同步、缓存、审计和 CI

原因是：

- 订阅生成逻辑是项目核心
- 后台只是配置入口，不应先于核心逻辑复杂化
- Cloudflare 资源绑定与部署应该建立在可运行代码之后

---

## 16. 风险与规避

### 风险 1：模板能力过于灵活导致系统失控

规避：

- 不支持运行任意 JS
- 只支持受控占位符与结构化渲染

### 风险 2：同步失败覆盖有效配置

规避：

- 保留最后一次成功快照
- 新结果校验失败时不覆盖现有快照

### 风险 3：数据库结构过早复杂化

规避：

- MVP 先保留必要表
- 高级特性通过新增表和字段扩展

### 风险 4：订阅缓存导致脏数据

规避：

- 明确失效触发点
- 对关键修改路径统一执行缓存清理

---

## 17. 当前建议的下一步

下一步不要直接铺太大，而是从最小可用工程开始：

1. 初始化根 `package.json` 和 workspace
2. 初始化 `apps/worker`
3. 初始化 `apps/web`
4. 初始化 `packages/shared` 与 `packages/core`
5. 编写第一版 migration

这样做的好处是：

- 每一步都可验证
- 不会因为前期文档过大而迟迟无法落地
- 后续可以严格按阶段推进

---

## 18. 本文档结论

SubForge 的正确落地路径不是“一次性做完整平台”，而是：

- 先做稳定的订阅生成能力
- 再做可操作的后台管理
- 再做同步、缓存和日志
- 最后做增强功能与自动化部署

该方案适合当前仓库从 0 到 1 开始建设，也适合后续逐步迭代到长期可维护版本。
