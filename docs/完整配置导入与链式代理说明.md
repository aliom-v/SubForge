# SubForge 完整配置导入与链式代理说明

## 1. 文档目的

本文档用于统一说明 SubForge 在以下三块能力上的当前状态、真实语义和使用边界：

1. 完整配置导入
2. `ssr` / `tuic` 协议支持
3. 链式代理与上游代理关系

这三块能力彼此相关：

- 完整配置导入会把 Clash / Mihomo YAML 与 sing-box JSON 中的节点提取为 SubForge 节点
- 导入时会尽量把配置里的链式关系保留为统一字段 `params.upstreamProxy`
- 后续订阅输出时，再按目标格式把这层关系还原为 Mihomo 的 `dialer-proxy` 或 sing-box 的 `detour`

如果不把这几层说清楚，很容易出现以下误解：

- “已经支持 Clash / sing-box 配置导入”被理解成“已经是完整配置编辑器”
- “节点已经导入成功”被理解成“配置里所有 group / route / graph 都被完整产品化管理”
- “页面上有链式代理”被理解成“已经有独立图模型、拖拽编辑器和链式校验引擎”

本文档的目标就是把这些边界讲透。

---

## 2. 一句话结论

当前版本已经支持：

- 导入 `vless://`、`trojan://`、`vmess://`、`ss://`、`ssr://`、`tuic://`、`hysteria2://` / `hy2://` 分享链接
- 导入整段 Base64 包装的分享链接订阅文本
- 导入完整的 Clash / Mihomo YAML 配置
- 导入完整的 sing-box JSON 配置
- 从完整配置中同时提取“节点清单 + 模板骨架”
- 在节点页可视化设置上游代理
- 在节点页查看链式代理拓扑
- 订阅渲染时把统一的 `params.upstreamProxy` 映射回目标格式

当前版本仍然不是：

- 完整配置编辑器
- 完整的 proxy-group / route 图形编辑器
- 独立的链式代理图数据库
- 自动持久化远程订阅并持续同步的节点源管理系统

所以更准确地说，当前实现的是：

- “完整配置导入”
- “链式代理可视化录入与校验”
- “导入配置到 SubForge 统一节点模型后的继续运营”

而不是“把原始 Clash / sing-box 配置文件原封不动变成可视化全量编辑器”。

## 3. 当前支持范围

### 3.1 分享链接导入支持

当前支持的分享链接协议：

- `vless://`
- `trojan://`
- `vmess://`
- `ss://`
- `ssr://`
- `tuic://`
- `hysteria2://`
- `hy2://`

这些能力同时覆盖：

- 本地粘贴多行分享链接导入
- 远程订阅 URL 预览导入
- Base64 包装订阅文本自动解包

### 3.2 完整配置导入支持

当前支持的完整配置输入：

- Clash / Mihomo YAML
- sing-box JSON

当前支持的完整配置输出结果：

- 节点预览
- 批量创建节点
- 模板骨架预览
- 直接创建模板
- 把模板载入模板创建表单后继续调整

### 3.3 协议向导支持

当前已提供结构化字段向导的协议：

- `vless`
- `trojan`
- `vmess`
- `ss`
- `ssr`
- `tuic`
- `hysteria2`

这意味着：

- 导入后的这些协议节点，不必完全回到原始 JSON 才能修改
- 常见字段可以在页面向导里直接调整
- 更复杂或当前未专门建模的字段，仍然继续通过 JSON 维护

### 3.4 订阅输出支持

当前支持的输出目标：

- `mihomo`
- `singbox`

链式代理映射规则：

- `params.upstreamProxy` -> Mihomo `dialer-proxy`
- `params.upstreamProxy` -> sing-box `detour`

## 4. 完整配置导入的真实语义

### 4.1 导入的不是“整份配置直接托管”

完整配置导入的真实行为是：

1. 解析原始配置
2. 抽取其中可识别的代理节点
3. 将这些节点转换为 SubForge 的统一节点模型
4. 保留原配置中的部分静态结构
5. 生成一个带模板插槽的模板骨架

这里最重要的点是：

- 节点会沉淀成 SubForge 的节点记录
- 模板会沉淀成 SubForge 的模板记录
- 导入后的后续管理，是围绕 SubForge 的统一模型继续进行

也就是说，导入完成以后，你管理的对象不再是“原始文件本身”，而是：

- 节点
- 模板
- 用户绑定
- 预览 / 公开订阅

### 4.2 Mihomo YAML 导入后会保留什么

Mihomo / Clash YAML 导入后，系统会：

- 从 `proxies` 中提取可识别代理节点
- 保留已有的 `proxy-groups`
- 保留已有的 `rules`
- 把 `proxies` 替换成模板插槽 `{{proxies}}`
- 如原配置缺少 `proxy-groups` 或 `rules`，会注入默认占位结构并给出 warning

如果代理节点里存在：

- `dialer-proxy`
- `dialerProxy`

则会在 SubForge 节点中统一写为：

```json
{
  "upstreamProxy": "上游节点名称"
}
```

### 4.3 sing-box JSON 导入后会保留什么

sing-box JSON 导入后，系统会：

- 从 `outbounds` 中提取可识别的代理 outbound
- 保留非代理类 static outbound，例如 `selector`、`urltest`、`direct`、`block`、`dns`
- 保留已有 `route.rules`
- 把动态代理节点部分替换成模板插槽

如果 outbound 中存在：

- `detour`

则会在 SubForge 节点中统一写为：

```json
{
  "upstreamProxy": "上游节点名称"
}
```

### 4.4 什么会被当成“可导入节点”

当前导入器会优先识别常见代理协议：

- `vless`
- `trojan`
- `vmess`
- `ss`
- `ssr`
- `tuic`
- `hysteria2`

这意味着：

- 这些协议会被提取成节点
- 不属于代理节点的配置块不会被当作节点导入
- 原配置中的静态结构会保留在模板侧，而不是硬塞进节点表

### 4.5 为什么这仍然不是完整配置编辑器

因为当前系统没有把这些对象全部做成独立的结构化后台实体：

- Mihomo `proxy-groups`
- sing-box `route.rules`
- sing-box `outbounds` 的全部静态图结构
- 更复杂的 graph / selector / provider / route policy

所以当前正确理解应当是：

- “支持从完整配置导入进入 SubForge”
- 而不是“后台已经能完整、无损、全图编辑所有配置元素”

## 5. `ssr` / `tuic` 当前支持到哪一层

### 5.1 `ssr`

当前 `ssr` 已支持：

- 手动录入
- 分享链接导入
- 远程订阅预览导入
- Clash / Mihomo 配置导入
- 结构化字段向导
- 订阅输出

当前常见字段：

- `credentials.cipher`
- `credentials.password`
- `credentials.protocol`
- `credentials.obfs`
- `params["protocol-param"]`
- `params["obfs-param"]`

### 5.2 `tuic`

当前 `tuic` 已支持：

- 手动录入
- 分享链接导入
- 远程订阅预览导入
- Clash / Mihomo 配置导入
- sing-box 配置导入
- 结构化字段向导
- 订阅输出

当前常见字段：

- `credentials.uuid`
- `credentials.password`
- `params.sni`
- `params.alpn`
- `params["congestion-controller"]`
- `params["udp-relay-mode"]`
- `params["disable-sni"]`
- `params.heartbeat`
- `params["request-timeout"]`
- `params["reduce-rtt"]`

### 5.3 仍需保守看待的地方

虽然 `ssr` / `tuic` 已经进入“可用”状态，但仍然不要误解成“协议全量产品化”。

当前仍建议保守处理这些场景：

- 罕见变体字段
- 很少用的扩展参数
- 与特定客户端强绑定的特殊配置块
- 需要完整图结构才能表达的复杂组合

遇到这些情况时，优先策略仍然是：

- 先导入
- 再核对导入后的 JSON
- 最后用预览验证生成结果

## 6. 链式代理当前的实现方式

### 6.1 当前统一字段

SubForge 目前不额外引入单独的数据表来存链式关系，而是统一用：

```json
params.upstreamProxy
```

来表示“当前节点的上游代理节点名称”。

这带来的好处是：

- 不需要变更数据库结构
- 可以让导入、手动编辑、订阅输出使用同一套字段
- 能在不破坏现有节点模型的前提下快速落地

### 6.2 导入阶段如何写入

导入时的统一规则：

- Mihomo `dialer-proxy` / `dialerProxy` -> `params.upstreamProxy`
- sing-box `detour` -> `params.upstreamProxy`

### 6.3 输出阶段如何恢复

渲染到不同目标时的恢复规则：

- 输出到 `mihomo` 时，`params.upstreamProxy` 会写成 `dialer-proxy`
- 输出到 `singbox` 时，`params.upstreamProxy` 会写成 `detour`

### 6.4 页面上的可视化能力

当前节点页提供两类链式代理能力：

1. 创建 / 编辑节点时，直接选择“上游代理”
2. 查看“链式代理拓扑”表

拓扑表会根据现有节点自动计算：

- 当前节点
- 上游代理
- 整条链路
- 当前状态

并直接提示这些常见问题：

- 上游节点缺失
- 上游节点名称重复
- 上游节点已禁用
- 出现循环链路

### 6.5 为什么当前按“节点名称”而不是 ID 建链

因为当前导入源和目标渲染格式里，本身更自然的关联键就是节点 tag / name。

这意味着当前链式代理实现更贴近配置世界的原始语义：

- Mihomo `dialer-proxy` 用的是名称
- sing-box `detour` 用的是 tag

但这也带来一个约束：

- 节点名称最好保持唯一

如果节点名称重复，当前拓扑表会直接标出问题，不建议继续依赖这种配置。

## 7. 推荐使用路径

### 7.1 如果你手里是完整 Mihomo / Clash 配置

推荐路径：

1. 进入“节点”页
2. 找到“导入完整配置”
3. 粘贴完整 YAML
4. 先看节点预览和 warning
5. 点击“批量创建 N 个节点”
6. 点击“创建 mihomo 模板”或“载入模板表单”
7. 到“用户”页完成节点绑定
8. 到“预览”页验证 `mihomo` 输出

### 7.2 如果你手里是完整 sing-box 配置

推荐路径：

1. 进入“节点”页
2. 找到“导入完整配置”
3. 粘贴完整 JSON
4. 先看节点预览和静态 outbound 保留情况
5. 点击“批量创建 N 个节点”
6. 点击“创建 singbox 模板”或“载入模板表单”
7. 到“用户”页绑定节点
8. 到“预览”页验证 `singbox` 输出

### 7.3 如果你只是想调链式代理

推荐路径：

1. 在“创建节点”或“编辑节点”里设置“上游代理”
2. 保存节点
3. 查看“链式代理拓扑”表是否有异常
4. 到“预览”页验证输出里的 `dialer-proxy` / `detour`

## 8. 常见误区

### 8.1 “导入完整配置”是不是等于以后都不用模板了

不是。

完整配置导入会帮你生成模板骨架，但 SubForge 的订阅输出仍然是：

- 用户
- 节点
- 模板
- 规则

这条统一编译链路。

也就是说，模板没有消失，只是创建模板的门槛降低了。

### 8.2 “导入了 sing-box 配置”是不是所有 static outbounds 都能在后台可视化编辑

不是。

当前系统能做的是：

- 保留它们
- 在模板里继续带着走

但不是把每一种 static outbound 都做成了独立页面控件。

### 8.3 “链式代理可视化”是不是已经支持拖拽图编辑

不是。

当前实现的是：

- 表单选择上游代理
- 自动计算链路
- 表格化拓扑校验

还不是拖拽式图编辑器。

### 8.4 “支持协议导入”是不是等于协议所有变体都已覆盖

不是。

当前正确理解是：

- 主路径常见字段已经可用
- 常见导入链路已打通
- 更复杂变体仍应结合 JSON 和预览验证

---

## 9. 当前仍未覆盖的边界

目前仍未完成的部分主要有：

- 真正意义上的完整配置编辑器
- Mihomo `proxy-groups` 的结构化可视化编辑
- sing-box `route.rules` / static outbounds 的结构化可视化编辑
- 拖拽式链式代理图编辑器
- 基于独立图模型的链式代理 schema
- 远程订阅 URL 持久化与定时同步
- `hysteria2` 多端口等更复杂变体的完整产品化支持

这些都是后续可继续增强的方向，但不影响当前版本已经能完成：

- 完整配置导入
- 新增协议导入
- 链式代理配置与输出

---

## 10. 建议和维护约定

如果后续继续扩这块能力，建议保持以下约定：

1. 统一节点模型继续以 `credentials` + `params` 为稳定边界
2. 链式代理继续先走 `params.upstreamProxy`，不要同时并行出第二套表达
3. 新协议进入“常见协议”列表前，至少同步补齐：
   - 导入解析
   - 向导
   - 输出
   - 测试
   - 文档
4. 如果后续真的要做完整配置编辑器，再单独升级为：
   - 模板图模型
   - group / route / outbound 的独立可视化实体
