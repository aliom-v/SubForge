# SubForge 第四批自动化验证执行清单

## 1. 文档目的

本文档用于把第四批测试范围固定为“同步失败路径与更真实的 Worker 请求级验证”，避免继续扩展到真实远程环境或全量路由覆盖。

本轮只做三类验证：

1. 上游异常导致的规则源同步失败
2. 上游内容为空或解析失败时的错误详情记录
3. `/api/rule-sources/:id/sync` 的请求级成功 / 失败语义

当前进展：`done`

- 已补规则源同步失败路径测试，覆盖上游非 2xx、空内容与解析失败
- 已补 `/api/rule-sources/:id/sync` 请求级成功 / 失败 / 404 语义测试
- 已补同步失败时对上游状态、字节数与失败原因的 details 保留

---

## 2. 本轮边界

### 2.1 本轮要补

- `apps/worker/src/sync.ts` 中同步失败分支的结果与 details 语义
- `apps/worker/src/index.ts` 中规则源手动同步路由的请求级行为

### 2.2 本轮不补

- 真实远程 HTTP 集成测试
- Cron 调度端到端测试
- 全量 Worker 路由回归
- Web 页面级测试
- 覆盖率门槛与报告平台

---

## 3. 具体执行项

### 3.1 同步失败路径

目标文件：

- `apps/worker/src/sync.ts`

本轮至少覆盖：

- `syncRuleSourceNow` 在上游返回非 2xx 时返回 `failed`，记录失败日志，且不误触发缓存失效
- `syncRuleSourceNow` 在上游内容为空时返回 `failed`，details 中保留上游状态、字节数与失败原因
- `syncRuleSourceNow` 在规则解析失败时返回 `failed`，并尽量保留本次请求已拿到的结构化 details

### 3.2 请求级同步路由

目标文件：

- `apps/worker/src/index.ts`

本轮至少覆盖：

- `POST /api/rule-sources/:id/sync` 在成功同步时返回 `success` 结果，并写入审计日志
- `POST /api/rule-sources/:id/sync` 在同步失败时仍返回结构化结果，且审计 payload 中包含失败状态
- `POST /api/rule-sources/:id/sync` 对不存在的规则源返回 `404`

### 3.3 错误详情写实化

目标文件：

- `apps/worker/src/sync.ts`
- `README.md`
- `docs/自动化验证与CI计划.md`

本轮要求：

- 如果测试暴露当前失败分支没有保留足够的 `details` 字段，应先把实现补齐，再补充文档说明
- 文档只写当前真实可验证的同步详情语义，不把理想中的错误展示提前写成已实现
