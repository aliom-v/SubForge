# 节点协议示例库

## 1. 文档目的

本文档只整理可复制的协议样例，面向运营或排障时的快速对照。

- 流程说明看 `docs/节点管理与订阅使用说明.md`
- “导入成功但托管订阅没变化”排查看 `docs/节点管理与订阅使用说明.md`

当前首批覆盖：

- `vless`
- `trojan`
- `vmess`
- `ss`
- `ssr`
- `tuic`
- `hysteria2`

## 5. `ss`

### 5.1 分享链接示例

```text
ss://YWVzLTI1Ni1nY206cGFzc3cwcmQ=@ss-01.example.com:8388?plugin=v2ray-plugin#SS%20Node
```

### 5.2 结构化字段示例

- 名称：`SS Node`
- 协议：`ss`
- 地址：`ss-01.example.com`
- 端口：`8388`
- 向导常见字段：`cipher`、`password`、`plugin`

### 5.3 原始 JSON 对照

`credentials`

```json
{
  "cipher": "aes-256-gcm",
  "password": "passw0rd"
}
```

`params`

```json
{
  "plugin": "v2ray-plugin"
}
```

### 5.4 当前限制

- 更复杂的 plugin 参数继续建议直接核对原始 JSON
- `params.plugin-opts` / `params.pluginOpts` 这类复杂字段当前没有单独向导

### 5.5 常见报错对照

常见会看到：

- `ss 节点需要 credentials.cipher 和 credentials.password`
  说明：当前主路径至少要求这两个字段同时存在
- `ss 节点的 params.plugin 必须是非空字符串`
  说明：如果你显式填写了 `plugin`，它不能是空字符串
- `ss 节点暂不支持 params.plugin-opts / params.pluginOpts 这类复杂 plugin 字段，请继续直接核对原始 JSON`
  说明：复杂 plugin 变体当前没有结构化支持，建议直接对照原始 JSON

## 6. `ssr`

### 6.1 分享链接示例

```text
ssr://c3NyLmV4YW1wbGUuY29tOjQ0MzphdXRoX2FlczEyOF9tZDU6YWVzLTI1Ni1jZmI6dGxzMS4yX3RpY2tldF9hdXRoOmNHRnpjM2N3Y21RPS8/cmVtYXJrcz1VMU5TSUU1dlpHVT0mcHJvdG9wYXJhbT1NVEF3T25KbGNHeGhZMlV0YldVPSZvYmZzcGFyYW09YzNWaUxtVjRZVzF3YkdVdVkyOXQ=
```

### 6.2 结构化字段示例

- 名称：`SSR Node`
- 协议：`ssr`
- 地址：`ssr.example.com`
- 端口：`443`
- 向导常见字段：`cipher`、`password`、`protocol`、`obfs`、`protocol-param`、`obfs-param`

### 6.3 原始 JSON 对照

`credentials`

```json
{
  "cipher": "aes-256-cfb",
  "password": "passw0rd",
  "protocol": "auth_aes128_md5",
  "obfs": "tls1.2_ticket_auth"
}
```

`params`

```json
{
  "protocol-param": "100:replace-me",
  "obfs-param": "sub.example.com"
}
```

### 6.4 当前边界

- 当前主路径会稳定保留 `protocol-param` / `obfs-param`
- `group` 这类更少见字段继续建议直接核对原始 JSON
- 分享链接导入、结构化向导和统一托管输出现在都已覆盖 `ssr`

### 6.5 常见报错对照

常见会看到：

- `ssr 节点需要 credentials.cipher、credentials.password、credentials.protocol 和 credentials.obfs`
  说明：当前主路径至少要求这四个核心字段同时存在
- `ssr 节点的 params["protocol-param"] 必须是非空字符串`
  说明：如果你显式填写了 `protocol-param`，它不能是空字符串
- `ssr 节点的 params["obfs-param"] 必须是非空字符串`
  说明：如果你显式填写了 `obfs-param`，它不能是空字符串

## 7. `tuic`

### 7.1 分享链接示例

```text
tuic://11111111-1111-1111-1111-111111111111:replace-me@tuic.example.com:443?sni=sub.example.com&alpn=h3&congestion_control=bbr&udp_relay_mode=native&zero_rtt_handshake=1#TUIC%20Node
```

### 7.2 结构化字段示例

- 名称：`TUIC Node`
- 协议：`tuic`
- 地址：`tuic.example.com`
- 端口：`443`
- 向导常见字段：`uuid`、`password`、`sni`、`alpn`、`insecure`、`congestion-controller`、`udp-relay-mode`、`request-timeout`、`reduce-rtt`

### 7.3 原始 JSON 对照

`credentials`

```json
{
  "uuid": "11111111-1111-1111-1111-111111111111",
  "password": "replace-me"
}
```

`params`

```json
{
  "sni": "sub.example.com",
  "alpn": "h3",
  "insecure": true,
  "congestion-controller": "bbr",
  "udp-relay-mode": "native",
  "request-timeout": 8000,
  "reduce-rtt": true
}
```

### 7.4 当前边界

- `tuic` 当前已覆盖分享链接导入、结构化向导和统一托管输出
- 完整配置导入时，`sing-box` 的 `tls.insecure` 与 Mihomo / Clash 的 `skip-cert-verify` 现在也会保留进统一托管输出
- 更少见的传输细节仍建议继续直接核对原始 JSON
- `zero_rtt_handshake=1` 会统一收敛到 `params["reduce-rtt"] = true`

### 7.5 常见报错对照

常见会看到：

- `tuic 节点需要 credentials.uuid 和 credentials.password`
  说明：当前主路径至少要求 `uuid` 和 `password`
- `tuic 节点的 params.alpn 必须是字符串或非空字符串数组`
  说明：`alpn` 可以是单个字符串，也可以是非空字符串数组
- `tuic 节点的 params.insecure 必须是布尔值`
  说明：`insecure` 只能写 `true/false`
- `tuic 节点的 params["request-timeout"] 必须是数字`
  说明：如果你显式填写了 `request-timeout`，它必须是数值

## 8. `hysteria2`

### 8.1 分享链接示例

```text
hysteria2://replace-me@hy2-01.example.com:8443?sni=sub.example.com&obfs=salamander&obfs-password=secret&alpn=h3#HY2%20Node
```

```text
hy2://replace-me@hy2-02.example.com:8443,9443?sni=sub.example.com#HY2%20Multi
```

### 8.2 结构化字段示例

- 名称：`HY2 Node`
- 协议：`hysteria2`
- 地址：`hy2-01.example.com`
- 端口：`8443`
- authority 多端口时：`port` 取首个端口，完整表达式保留到 `params.mport`
- 向导常见字段：`password`、`sni`、`obfs`、`obfs-password`、`alpn`、`insecure`
- 如果来自完整 `sing-box` 配置，`network: "tcp" | "udp"` 会统一收敛到 `params.network`

### 8.3 原始 JSON 对照

`credentials`

```json
{
  "password": "replace-me"
}
```

`params`

```json
{
  "sni": "sub.example.com",
  "obfs": "salamander",
  "obfs-password": "secret",
  "alpn": "h3"
}
```

### 8.4 当前边界

- `authority` 里的多端口现在会收敛成 `port + params.mport`；如果 query 里也写了 `mport`，则优先保留 query 值
- 重复的 `alpn` / `pinSHA256` query 会按字符串数组保留
- 完整 `sing-box` 配置导入时，`network` / `server_ports` / `hop_interval` / `up_mbps` / `down_mbps` 和 `tls.certificate_public_key_sha256` 现在也会保留进统一托管输出
- 未知 query 参数会直接报 unsupported，而不是静默忽略
- 更复杂组合继续建议对照原始 JSON 再保存

### 8.5 常见报错对照

常见会看到：

- `hysteria2 节点需要 credentials.password`
  说明：当前主路径至少要求存在 `password`
- `hysteria2 节点当前仅支持 params.obfs = "salamander"`
  说明：目前只收敛了 `salamander` 这条常见路径
- `hysteria2 分享链接包含当前不支持的参数: foo`
  说明：未知 query 参数会直接拒绝，不再静默忽略
- `hysteria2 节点的 params.insecure 必须是布尔值`
  说明：`insecure` 只能写 `true/false`

如果你碰到的是更复杂组合，建议同时对照：

- 当前分享链接
- 当前 `credentials` / `params` JSON
- `docs/节点管理与订阅使用说明.md`

## 9. 使用建议

- 能用结构化向导的协议，优先先填向导，再核对 JSON
- 导入成功后如需客户端直接使用，统一走这条主流程：导入 -> 调整节点 -> 使用当前启用节点生成托管 URL
- 如果你要排查“导入成功但订阅没变化”，直接看 `docs/节点管理与订阅使用说明.md`
