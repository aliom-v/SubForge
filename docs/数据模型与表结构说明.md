# SubForge 数据模型与表结构说明

## 1. 文档目的

本文档以**当前 migration 与共享类型**为准，说明：

- D1 中当前有哪些表
- 每张表的关键字段、用途和关系是什么
- 当前代码里常见的枚举值有哪些
- 哪些索引已经存在
- 这些表如何映射到当前 API / Worker 行为

设计类背景可配合阅读：

- `docs/实施方案.md`
- `docs/架构图与ER图.md`

## 2. 总览

当前 D1 由这 9 张核心表组成：

- `admins`
- `users`
- `nodes`
- `templates`
- `rule_sources`
- `rule_snapshots`
- `user_node_map`
- `sync_logs`
- `audit_logs`

其中：

- `admins` 负责后台管理员账号
- `users` 负责订阅用户与公开 token
- `nodes` 负责节点元数据
- `templates` 负责目标类型模板
- `rule_sources` / `rule_snapshots` 负责远程规则源与快照
- `user_node_map` 负责用户和节点绑定
- `sync_logs` / `audit_logs` 负责观测与追踪

## 3. 共享枚举与约束

当前共享层中实际使用的关键枚举：

- `SubscriptionTarget`
  - `mihomo`
  - `singbox`

- `UserStatus`
  - `active`
  - `disabled`

- `AdminStatus`
  - `active`
  - `disabled`

- `AdminRole`
  - `admin`

- `NodeSourceType`
  - `manual`
  - `remote`

- `RuleSourceFormat`
  - `yaml`
  - `json`
  - `text`

- `SyncLogStatus`
  - `success`
  - `failed`
  - `skipped`

## 4. 表结构说明

### 4.1 `admins`

用途：

- 存储后台管理员账号
- 登录时按 `username` 查询
- 会话 token 的 `sub` 指向管理员 `id`

主要字段：

- `id`：主键
- `username`：唯一用户名
- `password_hash`：密码哈希
- `role`：当前默认 `admin`
- `status`：当前默认 `active`
- `created_at`
- `updated_at`

关系：

- `audit_logs.actor_admin_id -> admins.id`

### 4.2 `users`

用途：

- 存储订阅用户
- `token` 用于公开订阅 `/s/:token/:target`

主要字段：

- `id`：主键
- `name`
- `token`：唯一公开订阅 token
- `status`：`active` / `disabled`
- `expires_at`：可空，到期时间
- `remark`：可空备注
- `created_at`
- `updated_at`

相关行为：

- 创建用户时自动生成高熵 token
- 支持 token 重置
- 用户资料或 token 变更会失效对应缓存

### 4.3 `nodes`

用途：

- 存储节点基础信息
- 供订阅编译时组装代理节点列表

主要字段：

- `id`：主键
- `name`
- `protocol`
- `server`
- `port`
- `credentials_json`：可空，认证信息对象
- `params_json`：可空，协议扩展参数
- `source_type`：默认 `manual`
- `source_id`：可空，预留远程来源标识
- `enabled`：布尔语义，SQLite 中存整数
- `last_sync_at`：可空
- `created_at`
- `updated_at`

相关行为：

- 当前支持单个创建、更新、删除与 JSON 批量导入
- 节点更新或删除会失效受影响用户缓存

### 4.4 `templates`

用途：

- 存储订阅模板
- 按 `target_type` 区分 `mihomo` 与 `singbox`

主要字段：

- `id`
- `name`
- `target_type`
- `content`
- `version`
- `is_default`
- `enabled`
- `created_at`
- `updated_at`

相关行为：

- 每个目标类型允许切换默认模板
- 默认模板变化会触发按目标类型的缓存失效

### 4.5 `rule_sources`

用途：

- 记录远程规则源配置
- 保存最近同步状态和失败次数

主要字段：

- `id`
- `name`
- `source_url`
- `format`
- `enabled`
- `last_sync_at`
- `last_sync_status`
- `failure_count`
- `created_at`
- `updated_at`

相关行为：

- 支持手动同步与 Cron 同步
- 同步后会写入 `sync_logs`
- 同步成功且内容变化时写入 `rule_snapshots`

### 4.6 `rule_snapshots`

用途：

- 保存规则源内容快照
- 用于去重、内容哈希比较与订阅编译读取

主要字段：

- `id`
- `rule_source_id`
- `content_hash`
- `content`
- `created_at`

关系：

- `rule_source_id -> rule_sources.id`

相关行为：

- 同一规则源 + 相同 `content_hash` 不会重复插入
- 编译订阅时会读取每个规则源最新快照

### 4.7 `user_node_map`

用途：

- 管理用户与节点的多对多绑定关系

主要字段：

- `id`
- `user_id`
- `node_id`
- `enabled`
- `created_at`

关系：

- `user_id -> users.id`
- `node_id -> nodes.id`
- `(user_id, node_id)` 唯一

相关行为：

- 当前通过“替换绑定列表”方式写入
- 绑定变化会触发用户缓存失效

### 4.8 `sync_logs`

用途：

- 记录规则源同步结果

主要字段：

- `id`
- `source_type`
- `source_id`
- `status`
- `message`
- `details_json`
- `created_at`

相关行为：

- 当前主要记录 `rule_source` 同步
- `details_json` 中会保存阶段、错误码、规则数、耗时、内容哈希等结构化信息

### 4.9 `audit_logs`

用途：

- 记录管理员的后台操作审计日志

主要字段：

- `id`
- `actor_admin_id`
- `action`
- `target_type`
- `target_id`
- `payload_json`
- `created_at`

关系：

- `actor_admin_id -> admins.id`

相关行为：

- 列表查询时会联表补出管理员用户名与资源展示名
- payload 中会附带请求上下文，如方法、路径、Ray ID、IP、国家、colo、UA；API 列表响应会再拆出 `requestMeta` 字段便于前端展示

## 5. 索引说明

当前 migration 中已显式创建这些索引：

- `idx_users_status`
- `idx_users_expires_at`
- `idx_nodes_enabled`
- `idx_nodes_protocol`
- `idx_nodes_source`
- `idx_templates_target_enabled`
- `idx_templates_target_default`
- `idx_rule_sources_enabled`
- `idx_rule_snapshots_hash`（唯一）
- `idx_rule_snapshots_source_created`
- `idx_user_node_map_user_id`
- `idx_user_node_map_node_id`
- `idx_sync_logs_source_type`
- `idx_sync_logs_created_at`
- `idx_audit_logs_actor_admin_id`
- `idx_audit_logs_target`

这些索引当前主要服务于：

- 用户状态与过期筛选
- 节点按启用状态与来源读取
- 模板按目标类型与默认模板读取
- 规则源按启用状态同步
- 快照去重与按时间读取最新快照
- 用户/节点绑定查询
- 同步日志与审计日志列表倒序读取

## 6. 关系图（逻辑）

可按下面方式理解当前核心关系：

```text
admins ──< audit_logs
users  ──< user_node_map >── nodes
rule_sources ──< rule_snapshots
```

而订阅编译阶段会把这些数据拼起来：

- `users`
- `user_node_map`
- `nodes`
- `templates`
- `rule_snapshots`

## 7. 数据到 API 的映射提示

当前共享类型与数据库表并不完全一一同名，但大体映射如下：

- `AdminRecord` ↔ `admins`
- `UserRecord` ↔ `users`
- `NodeRecord` ↔ `nodes`
- `TemplateRecord` ↔ `templates`
- `RuleSourceRecord` ↔ `rule_sources`
- `RuleSnapshotRecord` ↔ `rule_snapshots`
- `UserNodeBinding` ↔ `user_node_map`
- `SyncLogRecord` ↔ `sync_logs`
- `AuditLogRecord` ↔ `audit_logs`

需要注意：

- `enabled` 在 D1 中是整数，在 TypeScript 侧会映射为布尔值
- `credentials_json` / `params_json` / `details_json` / `payload_json` 会在 repository 层解析为对象
- `AuditLogRecord` 在 repository / API 侧还会补出 `targetDisplayName` 与 `requestMeta` 便于后台表格展示
- `templates.enabled` 会在 API 侧映射成 `status: enabled|disabled`

## 8. 当前边界

这份文档以当前仓库现状为准，因此：

- 只描述 `migrations/001_init.sql` 中已有结构
- 不包含未来可能新增的远程节点源专用表
- 不代表长期最终 schema，只代表当前实现

如果后续表结构演进，建议同步更新本文档、相关 API / 部署文档，并在 `CHANGELOG.md` 中记录变更。
