# SubForge 发布前检查清单

## 1. 文档目的

本文档用于把“部署完成后到底怎么做一轮最小验收”固定成一份可执行清单，避免每次发布前都重新口头整理。

如果你还没把环境跑起来，先看：

- `docs/部署指南.md`

如果你要核对接口、字段或报错语义，再看：

- `docs/API参考与接口约定.md`
- `docs/节点字段字典.md`
- `docs/常见错误与返回语义.md`

---

## 2. 适用时机

建议在下面这些时间点使用这份清单：

- 首次部署完成后
- 调整了模板、节点、绑定逻辑后
- 补了协议导入或字段校验后
- 准备对外演示或交付前

---

## 3. 基础环境检查

- [ ] `GET /health` 返回 `200`
- [ ] 返回体里能看到 `ok: true`
- [ ] Worker 当前已经绑定正确的 `DB` 和 `SUB_CACHE`
- [ ] `GET /api/setup/status` 能正常返回，不再出现 `no such table: admins`
- [ ] 如果当前环境是首次安装，`POST /api/setup/bootstrap` 能成功创建管理员
- [ ] 如果当前环境已初始化，`POST /api/admin/login` 能成功返回 Bearer token

验收口径：

- `health` 通，说明 Worker 基本可用
- `setup/status` 通，说明至少 migration 和数据库绑定没彻底跑偏
- 登录通，说明后台主链路已经可进入

---

## 4. 模板与基础数据检查

- [ ] 至少存在一套启用中的默认模板
- [ ] 目标类型至少覆盖你要验收的输出类型：`mihomo` 或 `singbox`
- [ ] 新建或编辑模板后，后台列表状态与默认标记符合预期
- [ ] 至少存在一个可用用户
- [ ] 至少存在一个启用节点
- [ ] 目标用户已经完成节点绑定

最小通过口径：

- 没有默认模板，后面的预览和公开订阅都不算验收通过
- 没有绑定节点，公开订阅不变化通常不是缓存问题，而是输入不完整

---

## 5. 节点导入与字段核对

### 5.1 分享链接或远程预览

- [ ] 如果你验收的是分享链接导入，至少成功导入一条 `vless://` / `trojan://` / `vmess://` / `ss://` / `hysteria2://` / `hy2://`
- [ ] 如果你验收的是订阅 URL 远程预览，`POST /api/node-import/preview` 返回 `200`
- [ ] 远程预览返回里能看到 `nodes` 和 `errors`
- [ ] 如果输入是 Base64 包装订阅文本，返回里的 `contentEncoding` 符合预期

### 5.2 metadata 核对

- [ ] `ss` 节点至少确认 `credentials.cipher` 与 `credentials.password`
- [ ] `hysteria2` 节点至少确认 `credentials.password`
- [ ] 如果有 `plugin` / `sni` / `obfs` / `obfs-password` / `alpn` / `pinSHA256`，确认它们落到了 `params`
- [ ] 如果出现字段校验错误，先对照 `docs/节点字段字典.md` 和 `docs/常见错误与返回语义.md`

最小通过口径：

- “导入成功”不等于 metadata 完整
- 结构化向导、分享链接导入和最终提交，归根结底都要落回 `credentials` / `params`

---

## 6. 预览与公开订阅验收

### 6.1 后台预览

- [ ] `GET /api/preview/:userId/:target` 返回 `200`
- [ ] 返回体里能看到 `cacheKey`、`mimeType`、`content`、`metadata`
- [ ] 第一次请求观察到 `x-subforge-preview-cache: miss`
- [ ] 不改任何数据再次请求，观察到 `x-subforge-preview-cache: hit`

### 6.2 公开订阅

- [ ] `GET /s/:token/:target` 返回 `200`
- [ ] 返回内容是订阅纯文本，不是 HTML，也不是错误 JSON
- [ ] 第一次请求观察到 `x-subforge-cache: miss`
- [ ] 不改任何数据再次请求，观察到 `x-subforge-cache: hit`

### 6.3 一致性

- [ ] 后台预览使用的用户、target 与公开订阅使用的 token、target 是同一组对象
- [ ] 如果你验收的是 `mihomo`，不要误拿 `singbox` 的模板或订阅地址做比较
- [ ] 预览内容和公开订阅内容至少在节点条目数量和核心配置上大体一致

---

## 7. 变更后缓存与回归检查

- [ ] 修改节点后再次请求预览，能看到结果变化
- [ ] 修改节点绑定后再次请求公开订阅，能看到结果变化
- [ ] 切换默认模板后，再次请求对应 target，结果能反映新模板
- [ ] 规则源启停或同步后，再次请求订阅，结果能反映最新规则

如果你观察到“第一次和第二次都完全没变化”，优先排查：

1. 变更是否真的写入成功
2. 用户和 target 是否选对
3. 是否访问了旧 token 或旧订阅地址
4. 客户端是否缓存了旧结果

---

## 8. 失败时优先跳转

- 页面卡在“加载安装状态”：看 `docs/部署指南.md`
- 节点导入成功但输出没变化：看 `docs/导入成功但订阅没变化排障指南.md`
- `ss` / `hysteria2` 字段校验失败：看 `docs/节点字段字典.md`
- 不确定错误码和状态码含义：看 `docs/常见错误与返回语义.md`
- 不确定接口本身该怎么请求：看 `docs/API参考与接口约定.md`

---

## 9. 最小发布通过口径

只有同时满足下面这些条件，才建议视为“这轮发布前最小验收通过”：

1. 健康检查、setup/status、管理员登录都正常
2. 至少一个默认模板、一个用户、一个节点和一组绑定是有效的
3. 后台预览能返回正确内容，并且能看到 `miss -> hit`
4. 公开订阅能返回正确内容，并且能看到 `miss -> hit`
5. 节点导入、字段校验、缓存行为没有出现明显和文档相反的语义
