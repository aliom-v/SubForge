# SubForge 单用户托管模式需求

## 1. 文档目的

本文档用于把当前版本真正要服务的目标收拢成一份可执行范围说明，避免继续把仓库写成“完整多用户后台平台”。

当前方向不是做机场后台全家桶，而是先把下面这条链路做扎实：

1. 部署到 Cloudflare
2. 绑定自己的域名
3. 导入节点或上游配置
4. 直接得到可被客户端拉取的托管 URL

---

## 2. 当前需求

### 2.1 核心目标

- 只服务一个管理员 / 一套个人托管订阅
- 主界面只围绕三种导入格式展开：
  - 节点文本导入
  - 订阅 URL 解析
  - 导入完整配置
- 导入后直接生成可访问的托管 URL
- 让 Mihomo / sing-box 客户端直接通过 URL 拉取使用
- 不依赖传统常驻服务器，优先使用 Cloudflare Worker + D1 + KV

### 2.2 解析要求

当前解析链路必须优先保证下面几类输入足够稳定：

- 分享链接
  - `vless://`
  - `trojan://`
  - `vmess://`
  - `ss://`
  - `ssr://`
  - `tuic://`
  - `hysteria2://` / `hy2://`
- Base64 包装的订阅文本
- JSON 节点清单
- 完整 Mihomo / Clash YAML
- 完整 sing-box JSON

要求不是“支持了名字就算完成”，而是：

- 能识别出输入类型
- 能给出可读的解析错误
- 能把成功解析出的节点稳定导入
- 能在导入后直接进入托管输出链路

### 2.3 使用体验要求

- 主界面默认只讲单用户托管模式，不要求用户理解模板、规则集、用户绑定这些内部对象
- 托管 URL 必须基于当前站点 `origin` 生成，方便直接绑定自定义域名使用
- 导入后的反馈必须足够直白：
  - 识别到了什么格式
  - 解析出了多少节点
  - 哪些内容失败了
  - 当前托管 URL 是什么

---

## 3. 当前完成度

### 3.1 已完成

- Cloudflare Worker + D1 + KV 的无传统服务器部署形态
- 后台首次安装、管理员登录、同域部署
- 三种主入口：
  - 节点文本导入
  - 订阅 URL 解析
  - 导入完整配置
- 单用户托管 URL 自动生成
- 上游订阅 URL 可持久化为自动拉取任务
- Cron 自动定时拉取上游订阅并刷新对应节点
- 公开订阅输出：
  - `/s/:token/mihomo`
  - `/s/:token/singbox`
- 链式代理字段映射：
  - `dialer-proxy` / `detour` -> `params.upstreamProxy`
- 基础缓存、频控、审计

### 3.2 部分完成

- 完整配置导入
  - 当前已经能提取节点，并尽量把关键结构带入托管输出
  - 但还不是完整配置编辑器
- 解析鲁棒性
  - 主流输入已覆盖
  - 仍应继续围绕真实输入样本补强边角协议和报错质量

### 3.3 仍未完成

- 通用第三方订阅输出格式
- 多用户、多租户、计费与复杂权限

---

## 4. 当前非目标

下面这些能力不是这个阶段的主线，不应继续占用首页心智：

- 手工维护模板和规则集
- 复杂的用户管理和节点绑定运营
- 完整的配置图形化编辑器
- 机场式多用户套餐与计费系统
- 代理内核本身的运行、测速或可用性探测

说明：

仓库内部仍保留用户、模板、规则源等数据结构，是因为当前托管输出仍需要这些底层对象完成编译与缓存；但它们应该被看作“内部实现细节”，不是单用户模式的主操作面。

---

## 5. 执行原则

- 新功能优先围绕三种导入入口和托管 URL 展开
- 文档优先写“当前真实可用路径”，不要再扩写高级后台故事
- 任何解析增强都优先解决真实输入样本，而不是先做抽象设计
- 如果某个能力暂时没有做自动化，就明确写成“手动”或“一次性预览”，不要模糊成“像是已经支持”
- 自动拉取能力默认按 `wrangler.toml` 当前 Cron 执行；如果要改频率，应先改部署配置，再改文档说明
