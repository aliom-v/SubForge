# SubForge 限流与安全策略

## 1. 文档目的

本文档只保留“策略层”信息，重点回答这些问题：

- 哪些接口已经做了限流
- 当前限流按什么维度统计
- 默认阈值是多少
- 应该如何调参
- 当前安全边界和后续增强方向是什么

为避免和协议文档重复，以下内容统一放到 `docs/API错误码与响应头说明.md`：

- 统一错误包裹结构
- `Authorization: Bearer` 的协议说明
- `retry-after`、`x-subforge-rate-limit-*`、`x-subforge-cache-key` 的响应头速查

本文档描述的是**当前仓库实现**，不是理想化方案。

## 2. 当前已实现的策略

### 2.1 管理员登录失败限流

作用接口：

- `POST /api/admin/login`

当前行为：

- 仅统计**登录失败**请求
- 统计维度为**客户端 IP**
- 同一 IP 在窗口期内失败次数达到阈值后，后续登录会直接返回 `429 TOO_MANY_REQUESTS`
- 一旦某次登录成功，会清空该 IP 当前的失败计数

当前默认值：

- `ADMIN_LOGIN_RATE_LIMIT_WINDOW_SEC=600`
- `ADMIN_LOGIN_RATE_LIMIT_MAX_ATTEMPTS=5`

实现细节：

- 计数存储在 `SUB_CACHE` KV 中
- key 前缀为 `rate:admin-login`
- 当前实现按 IP 聚合，不再额外区分用户名
- Cloudflare 环境优先使用 `cf-connecting-ip`，本地或代理环境回退到 `x-forwarded-for`

### 2.2 公开订阅频控

作用接口：

- `GET /s/:token/:target`

当前行为：

- 每次公开订阅请求都会先经过频控检查
- 统计维度为 **token + target + 客户端 IP**
- 频控发生在缓存读取之前，因此**缓存命中请求也会计数**
- 超限时返回 `429 TOO_MANY_REQUESTS`

当前默认值：

- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC=60`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS=60`

实现细节：

- 计数存储在 `SUB_CACHE` KV 中
- key 前缀为 `rate:subscription`
- 频控维度会区分订阅 token、目标类型和客户端 IP

## 3. 可观测行为

### 3.1 登录限流

你可以从这些行为判断登录限流是否生效：

- 登录成功时会清空失败计数，并返回 `x-subforge-rate-limit-cleared: true`
- 登录失败但未超限时，会继续返回当前额度相关头部
- 超限时会返回 `429` 与 `TOO_MANY_REQUESTS`
- 超限时可观察 `retry-after`

### 3.2 公开订阅频控

公开订阅在这些情况下都会带限流相关头：

- 缓存命中
- 缓存未命中但编译成功
- token 不存在或模板不可用导致的 `404`
- 订阅编译失败导致的 `400`
- 频控超限导致的 `429`

如果你只需要确认线上返回了哪些头、错误体长什么样，请直接看 `docs/API错误码与响应头说明.md`。

## 4. 配置项

当前与限流直接相关的环境变量如下：

- `ADMIN_LOGIN_RATE_LIMIT_WINDOW_SEC`
- `ADMIN_LOGIN_RATE_LIMIT_MAX_ATTEMPTS`
- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS`

默认值已写入：

- `.env.example`
- `wrangler.toml`

如果你在 Cloudflare Dashboard 中覆盖这些值，应与仓库文档保持一致，避免本地与线上行为不一致。

## 5. 调参建议

### 5.1 管理员登录失败限流

建议：

- 个人自用或仅少数管理员场景，可保持默认值或进一步收紧
- 如果后台只允许内网 / VPN / Zero Trust 访问，可把最大失败次数降到 `3`
- 如果管理员经常使用密码管理器自动重试，避免把窗口设得过短

一个偏保守的例子：

- `ADMIN_LOGIN_RATE_LIMIT_WINDOW_SEC=900`
- `ADMIN_LOGIN_RATE_LIMIT_MAX_ATTEMPTS=3`

### 5.2 公开订阅频控

建议：

- 默认值适合一般个人订阅客户端的轮询频率
- 如果同一个 token 会被多设备或聚合器频繁拉取，可适当调大阈值
- 如果遇到异常刷取、探测或 token 泄漏风险，可先降低窗口内次数并观察日志

一个偏宽松的例子：

- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC=60`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS=180`

一个偏保守的例子：

- `SUBSCRIPTION_RATE_LIMIT_WINDOW_SEC=60`
- `SUBSCRIPTION_RATE_LIMIT_MAX_REQUESTS=30`

## 6. 当前边界与安全建议

当前实现刻意保持轻量，因此需要注意这些边界：

- 只覆盖了**管理员登录失败限流**与**公开订阅频控**
- `preview` 接口当前**没有单独限流**
- 其余后台写接口当前**没有额外频控**，主要依赖管理员鉴权
- 登录限流当前按 **IP** 聚合；多个管理员共用同一出口 IP 时会共享失败额度
- 公开订阅频控当前按 **token + target + IP** 统计；同一出口 IP 下的多个客户端会共享该 token 的额度
- 计数依赖 Cloudflare KV，属于**轻量、近实时、非强一致**方案，更适合作为“削峰 / 防滥用”而不是严格配额器

因此它不能替代这些措施：

- token 保密与轮换
- 高强度管理员密码
- Cloudflare WAF / Rate Limiting Rules
- Cloudflare Zero Trust
- 审计日志与异常登录告警

## 7. 运维与验证建议

### 7.1 本地验证登录限流

```bash
curl -i http://127.0.0.1:8787/api/admin/login \
  -H 'content-type: application/json' \
  -d '{"username":"admin","password":"wrong-password"}'
```

建议观察：

- `401` 时的 `x-subforge-rate-limit-*`
- 超限后的 `429`
- 超限后的 `retry-after`

### 7.2 本地验证公开订阅频控

```bash
curl -i http://127.0.0.1:8787/s/your-token/mihomo
```

建议观察：

- `x-subforge-rate-limit-limit`
- `x-subforge-rate-limit-remaining`
- `x-subforge-rate-limit-reset`
- 超限时的 `retry-after`

### 7.3 线上联动建议

如果准备上线到真实用户环境，建议把当前轻量限流与这些措施组合使用：

- Cloudflare WAF / Rate Limiting Rules
- Cloudflare Zero Trust（后台入口）
- 更细的审计日志与异常登录告警
- token 泄漏后的快速失效机制
- 管理员入口与公开订阅入口的分域或额外访问控制

## 8. 后续可增强方向

推荐后续继续演进：

- 给 `preview` 与关键后台写接口补充独立频控
- 把登录限流结果写入审计日志或安全日志
- 增加基于管理员账号、国家 / 地区、UA 指纹的更细粒度策略
- 接入 Cloudflare 原生 Rate Limiting / Turnstile / WAF 规则
- 为超限与异常登录增加通知能力
