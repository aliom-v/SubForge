## 4. 当前问题清单

### P0. 安装链路与文档不一致

- 状态：`done`
- 严重度：`high`
- 现象：
  - 文档写明可先执行 `npm install`
  - 当前核对环境执行 `npm install` 时返回 `Unsupported URL Type "workspace:"`
- 影响：
  - 新环境无法按文档完成依赖安装
  - `typecheck` / `build` / `dev` 无法作为“已验证能力”对外陈述
- 建议修复：
  - 已把内部 workspace 依赖改为 npm 可安装的固定版本，并补充 `packageManager` 元数据
  - 已重新验证 `npm install`、`npm run typecheck`、`npm run build`
  - 已生成并保留 `package-lock.json`，减少新环境安装漂移
- 完成标准：
  - 新环境按文档能完成安装、类型检查和构建

### P1. 审计日志记录了明文用户 token

- 状态：`done`
- 严重度：`high`
- 现象：
  - 用户重置 token 时，旧 token 和新 token 会被写入审计日志
  - 管理入口中的审计日志视图会直接展示对应 payload
- 影响：
  - 有效订阅凭证暴露给管理员
  - 敏感凭证长期留存在数据库日志中
- 建议修复：
  - 已改为只记录 token 已重置事件，不再写入明文 token
  - 审计日志 API 输出也会对旧日志中的敏感字段做脱敏
  - 如有历史真实数据，后续可补数据库清理脚本
- 完成标准：
  - 审计日志与前端页面不再出现完整 token

### P1. 规则源启用状态变更未统一触发缓存失效

- 状态：`done`
- 严重度：`high`
- 现象：
  - 规则源启用 / 禁用会影响订阅编译输入
  - 当前更新规则源启用状态后，没有统一清理相关订阅缓存与预览缓存
- 影响：
  - 已禁用或启用的规则源，在 TTL 窗口内可能无法及时反映到输出结果
  - 与实施方案中“关键变更触发缓存失效”的原则不一致
- 建议修复：
  - 已在规则源 `enabled` 变化时统一清理用户缓存
  - 后续可继续补更细粒度的失效与相关回归测试
- 完成标准：
  - 规则源启停后，预览与公开订阅都能稳定反映最新状态

### P1. 禁用管理员仍可完成登录

- 状态：`done`
- 严重度：`medium`
- 现象：

  - 前端会出现“登录成功后立即失效”的体验问题
- 建议修复：
  - 已在登录阶段直接拦截禁用管理员
  - 后续可再决定是否继续收敛为统一认证错误文案
- 完成标准：

### P2. 公开订阅缓存命中时未重新校验用户状态与到期时间

- 状态：`done`
- 严重度：`medium`
- 现象：
  - 用户状态与到期时间校验主要发生在编译阶段
  - 公开订阅缓存命中时会直接返回缓存内容
- 影响：
  - 用户自然到期或状态变化后，在缓存 TTL 内仍可能拉到旧订阅
  - 与实施方案中的“先校验、再读缓存”描述不一致
- 建议修复：
  - 已在公开订阅接口中改为“先校验、再读缓存”
  - 缓存命中前会先校验 token 对应用户是否存在、是否启用、是否过期
  - 对于失效 token / 禁用用户 / 已过期用户，会顺手清理旧缓存 key，避免 stale 内容继续命中
- 完成标准：
  - 用户状态和到期变化不会再被旧缓存掩盖
