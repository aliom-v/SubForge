# SubForge 托管订阅与模板治理重构方案

## 1. 背景

当前仓库已经具备：

- 节点导入、完整配置导入、远程同步源同步
- 自动托管用户与托管 URL 生成
- `mihomo` / `singbox` 双输出
- 链式代理基础校验与节点拓扑展示

但线上已经暴露出两个直接风险：

1. 默认模板如果混入静态节点，最终托管订阅会重复节点或输出脏配置
2. 当前启用节点与托管绑定节点分离时，用户会看到“页面节点对了，订阅里却少节点”

这两个问题本质上都属于“模板骨架、节点数据、托管绑定、最终渲染”边界不够清晰。

---

## 2. 本次重构目标

这轮治理目标不是做通用配置编辑器，而是把托管主链路收紧成稳定生产路径：

1. 外部配置不能直接变成生产默认模板
2. 托管模板必须是“系统管理的动态骨架”
3. 公开订阅只能从“有效绑定节点 + 有效模板”生成
4. 坏模板不能继续污染线上托管输出
5. 链式代理能力继续保留，但不允许破坏订阅可用性

---

## 3. 设计原则

### 3.1 模板和节点解耦

- 模板负责骨架、规则、代理组结构
- 节点负责 `proxies` / `outbounds`
- 默认模板不允许内嵌生产节点数据

### 3.2 托管输出只认已绑定节点

- 页面上的“当前启用节点”只是候选集合
- 实际公开订阅只认托管用户当前绑定集合
- 绑定更新必须是显式动作，且要可诊断

### 3.3 先兼容，后收口

- 旧模板先尽量自动净化
- 无法安全净化时回退到系统默认骨架
- 不在第一阶段直接大改数据库结构

### 3.4 线上优先可用性

- 不允许坏模板继续输出重复节点
- 后续阶段引入“最后一次成功快照”
- 发布前统一体检，失败不覆盖旧结果

---

## 4. 现状判断

### 4.1 已确认问题

1. `mihomo` 渲染当前只会净化静态 `proxy-groups` 引用，不会主动清掉静态 `proxies`
2. 自动托管模板当前会优先复用导入配置或既有模板内容，容易把历史坏模板继续带下去
3. 公开订阅按托管用户绑定节点生成，因此“当前节点列表”和“实际输出节点”天然可能不同步

### 4.2 风险来源

- 导入整份外部配置时，模板继承过于宽松
- 历史模板可能包含静态节点
- 重新导入/同步节点后，用户未重新生成托管 URL

---

## 5. 目标架构

建议把当前主链路拆成 4 层：

1. 节点数据层
2. 模板骨架层
3. 托管绑定层
4. 最终订阅渲染层

### 5.1 节点数据层

职责：

- 保存节点协议、地址、端口、凭据、附加参数
- 保存链式代理字段
- 不承担模板语义

### 5.2 模板骨架层

职责：

- 保存规则、DNS、代理组结构、静态业务骨架
- 明确动态插槽：
  - `{{proxies}}`
  - `{{proxy_groups}}`
  - `{{rules}}`
  - `{{outbounds}}`

约束：

- 默认 `mihomo` 模板不允许保留静态 `proxies`
- 默认 `singbox` 模板后续要收紧静态节点型 `outbounds`

### 5.3 托管绑定层

职责：

- 表达“这个托管用户当前到底绑定了哪些节点”
- 与“当前启用节点列表”分离
- 支持诊断差异

### 5.4 渲染层

职责：

- 从托管绑定中取节点
- 从默认模板中取骨架
- 渲染为 `mihomo` / `singbox`
- 后续支持发布前体检和最后成功快照

---

## 6. 分阶段方案

## 6.1 第一阶段：模板止血

目标：

- 先阻止坏模板继续把静态节点带进托管输出

改动：

1. 新增 `mihomo` 托管模板规范化逻辑
2. 自动托管模板更新时，优先把 `proxies` 收敛为动态插槽
3. 历史 `mihomo` 托管模板如果可解析，则自动净化
4. 历史模板如果不可安全解析，则回退到系统默认托管骨架
5. 渲染前再次净化静态 `proxies`

完成标准：

- 自动托管 `mihomo` 默认模板不会继续带静态节点
- 托管生成流程可自动修复大多数历史坏模板
- 重复节点问题不再由模板层继续放大

## 6.2 第二阶段：托管订阅诊断

目标：

- 让“少节点、重复节点、绑定未同步”可直接看见

改动：

1. 托管卡片增加诊断区
2. 展示：
   - 当前启用节点数
   - 当前托管绑定节点数
   - 预览输出节点数
   - 差异节点列表
   - 重复节点名
3. 增加“一键重新绑定当前启用节点”

完成标准：

- 用户不再只能靠导入失败倒推问题

## 6.3 第三阶段：模板治理收口

目标：

- 不再让外部配置直接充当生产模板

改动：

1. 导入完整配置时只吸收结构，不直接继承生产模板
2. `mihomo`：
   - `proxies` 永远动态
   - `🚀 节点选择` / `⚡ 自动选择` 系统生成
3. `singbox`：
   - 节点型 `outbounds` 后续系统化收口
4. 后台保存模板时增加强校验

完成标准：

- 默认模板与导入配置快照彻底分层

## 6.4 第四阶段：链式代理模型升级

目标：

- 从“按名字引用上游”迁到“按稳定标识引用上游”

改动：

1. 引入稳定链路字段
2. 前端编辑器继续展示节点名，但后端存节点 ID / 稳定组 key
3. 兼容旧 `params.upstreamProxy`
4. 增加改名后的自动迁移或读时转换

完成标准：

- 节点改名不再破坏链路

## 6.5 第五阶段：发布与快照

目标：

- 线上订阅改成“通过体检才发布”

改动：

1. 生成前执行托管体检
2. 失败时不覆盖旧结果
3. 成功时发布新快照
4. 公开订阅读取最后一次成功快照

完成标准：

- 坏配置不会直接把用户的现有 URL 弄坏

---

## 7. 第一阶段具体实施口径

本轮代码先只落第一阶段，具体以这几个动作作为交付口径：

1. `mihomo` 自动托管模板统一走动态 `proxies`
2. 历史托管模板如果仍带静态 `proxies`，生成时自动净化
3. 无法解析的历史 `mihomo` 托管模板回退到内置骨架
4. 补回归测试，覆盖模板净化与订阅输出

本轮暂不做：

- `singbox` 静态 `outbounds` 精细治理
- 托管诊断 UI
- 最后成功快照
- 链式代理 ID 化迁移

---

## 9. 风险与规避

### 风险 1：历史模板结构过于脏，无法安全解析

规避：

- 第一阶段对 `mihomo` 不强行保留坏结构
- 解析失败时回退到系统默认骨架

### 风险 2：模板净化后业务组引用变化

规避：

- 仅先净化 `proxies`
- `proxy-groups` 继续保留，并沿用现有引用净化逻辑

### 风险 3：用户以为“启用节点”就是“已发布节点”

规避：

- 第二阶段补托管诊断卡片
- 页面显式区分“当前启用”和“当前托管绑定”
