# SubForge 排障与常见问题

## 1. 文档目的

本文档只保留“症状驱动”的 FAQ，集中回答这些高频问题：

- 本地为什么跑不起来
- Cloudflare 部署后为什么页面或 API 异常
- 为什么会出现 `401`、`403`、`404`、`429`、`Unexpected token '<'`
- 为什么规则同步成功了，但订阅内容看起来没变化

日常巡检、发版前后检查和告警建议统一放到 `docs/运维Runbook与告警处理.md`，避免和 FAQ 重复维护。

如果你要看协议细节或安全策略，请配合这些文档一起看：

- `docs/API错误码与响应头说明.md`
- `docs/限流与安全策略.md`
- `docs/部署指南.md`

## 2. 遇到问题先做 4 个快速确认

在深入排查前，先确认：

1. 先看 HTTP 状态码，再看 `error.code` 或关键响应头
2. `wrangler.toml` 是否仍保留 `run_worker_first = true`
3. `DB` / `SUB_CACHE` / `ADMIN_JWT_SECRET` 是否已正确绑定
4. 先跑一轮 `npm run test:smoke`，不要在基础环境异常时直接深挖业务数据

## 3. 本地开发常见问题

### 3.1 `npm run dev:worker` 启动失败

优先检查：

- Node.js 是否为 20+
- npm 是否为 10+
- 是否已经执行 `npm ci`（如在调整依赖，再执行 `npm install`）
- 本地是否有可用的 `wrangler`

如果 `smoke` 没通过，先修复基础环境，再继续看 Worker 运行问题。

### 3.2 `npm run dev:web` 打开后接口报错

先看这些点：

- Worker 是否已经启动在 `http://127.0.0.1:8787`
- `apps/web/vite.config.ts` 的 `/api`、`/s`、`/health` 代理是否生效
- 是否错误设置了 `VITE_API_BASE_URL`

常见表现：

- 登录时报网络错误
- 页面能打开，但接口一直 404 / 500
- 实际请求没打到本地 Worker

### 3.3 首次安装向导不可用

通常先确认：

- `GET /api/setup/status` 返回的 `initialized` 是否已经为 `true`
- D1 migration 是否已执行
- D1 中是否已经存在管理员记录

如果已经完成初始化，安装向导再次执行会被拒绝，这是预期行为。

## 4. Cloudflare 部署常见问题

### 4.1 页面报 `Unexpected token '<'`

这是最典型的症状之一，通常表示：

- 你以为拿到的是 JSON
- 实际拿到的是 HTML，通常是 SPA 的 `index.html`

优先检查：

1. `wrangler.toml` 是否启用了 `run_worker_first = true`
2. 当前线上部署是否真的使用了这版配置重新部署
3. `/api/*` 请求是否被错误回退到了静态页面
4. 浏览器网络面板里的 `content-type` 是否为 `text/html`

### 4.2 Cloudflare 没自动创建 D1 / KV

当前仓库依赖 Wrangler `4.45.0+` 的自动 provision 行为，但不同账号环境下不一定稳定。

如果没自动创建，手动补齐：

- `DB` → D1
- `SUB_CACHE` → KV

然后重新部署。

### 4.3 部署成功但后台登录失败

优先排查：

- `ADMIN_JWT_SECRET` 是否已配置
- D1 中是否真的有管理员记录
- 当前登录密码是否和安装向导 / seed 创建时一致
- 是否触发了登录失败限流

如果接口返回 `429`，重点观察：

- `retry-after`
- `x-subforge-rate-limit-remaining`
- `x-subforge-rate-limit-reset`

## 5. 数据与订阅问题

### 5.1 预览接口返回 `preview data not found`

这通常说明以下至少一项缺失：

- 用户不存在
- 用户没有可用节点
- 对应目标类型没有默认模板
- 当前编译上下文无法构建

建议检查：

- 用户是否存在且状态正常
- 用户是否绑定了启用状态的节点
- 对应 `mihomo` / `singbox` 默认模板是否存在且启用

### 5.2 公开订阅返回 `subscription token or template not found`

这通常意味着：

- token 不存在
- token 对应用户不可用
- 当前目标类型缺失可用模板

建议先排查：

- token 是否是最新值，尤其是刚重置过 token 时
- 用户状态是否为 `active`
- 默认模板是否存在且目标类型匹配

### 5.3 数据改了但订阅内容没变化

先区分两种情况：

1. 实际编译内容没变
2. 缓存还没失效，或尚未按最新数据重建

当前系统中：

- 用户、token、绑定变化会触发用户级缓存失效
- 节点变更会失效受影响用户缓存
- 模板变更会按目标类型失效缓存
- 后台支持手动执行缓存重建

建议重点看：

- `x-subforge-cache` 或 `x-subforge-preview-cache` 是 `hit` 还是 `miss`
- `x-subforge-cache-key` 是否符合当前用户 / 目标
- 规则源同步结果是否为 `skipped`

### 5.4 规则源同步后看起来没有更新

先看同步结果和同步日志：

- 是否是 `success`
- 是否是 `skipped`
- `details` 中的 `ruleCount`、`contentHash`、`reason` 是什么

如果是 `skipped`，通常说明规则内容在归一化后与上次相同，不一定是错误。

## 6. 认证与权限问题

### 6.1 为什么返回 `401 UNAUTHORIZED`

最常见原因：

- 没有带 Bearer token
- Bearer token 已失效或不合法
- 管理员登录用户名 / 密码错误

建议：

- 先确认请求头里是否有 `authorization: Bearer ...`
- 如果是登录接口，再看是不是用户名或密码输错
- 如果是连续登录失败后出现 `429`，转到限流问题排查

### 6.2 为什么返回 `403 FORBIDDEN`

当前常见触发：

- 管理员账号状态不可用
- 初始化已经完成却再次调用 bootstrap

这类问题通常不是“没登录”，而是“当前操作不被允许”。

### 6.3 登录为什么突然变成 `429`

当前登录限流只统计**失败登录**，并按 **IP** 聚合。

这意味着：

- 你自己连续输错多次会被限流
- 多个管理员共用同一出口 IP 时，也会共享失败额度

排查重点：

- `retry-after`
- `x-subforge-rate-limit-reset`
- `x-subforge-rate-limit-remaining`
- 最近是否频繁输错密码

### 6.4 公开订阅为什么返回 `429`

当前公开订阅按 **token + target + IP** 统计。

常见场景：

- 某个客户端轮询过快
- 多设备共用同一 token 且走同一出口 IP
- 外部聚合器或探测器频繁请求

排查重点：

- `x-subforge-rate-limit-remaining`
- `x-subforge-rate-limit-reset`
- `retry-after`
- 是否有异常外部拉取来源

更多策略细节可直接看 `docs/限流与安全策略.md`。

## 7. 缓存与响应头排查技巧

### 7.1 预览接口关注这些头

- `x-subforge-preview-cache`
- `x-subforge-cache-key`
- `x-subforge-cache-scope`

### 7.2 公开订阅接口关注这些头

- `x-subforge-cache`
- `x-subforge-cache-key`
- `x-subforge-cache-scope`
- `x-subforge-rate-limit-limit`
- `x-subforge-rate-limit-remaining`
- `x-subforge-rate-limit-reset`
- `retry-after`（仅 `429`）

### 7.3 为什么这些头重要

它们能帮你快速判断：

- 问题是缓存命中导致的旧数据
- 还是编译链路本身失败
- 还是频控拦截
- 还是请求压根没打到正确接口

## 8. 建议的排障顺序

遇到问题时，推荐按这个顺序：

1. 先看 HTTP 状态码
2. 再看 `error.code` 或关键响应头
3. 再确认是否命中缓存 / 限流
4. 再确认 D1 / KV / Secret / vars 绑定
5. 最后再检查具体业务数据（用户、节点、模板、规则源）
