# SubForge 高级部署与数据维护

本文档只保留高级部署、GitHub Actions、D1 备份恢复与对象存储归档这类内部运维内容。

如果你只是自己部署、自己使用，通常先看：

- `README.md`
- `docs/单用户托管模式需求.md`
- `docs/节点管理与订阅使用说明.md`
- `docs/部署指南.md`

## 1. 什么时候再看这篇

只有在下面这些场景，再进入这篇：

- 你要把部署接到 GitHub Actions
- 你要做 production 备份、恢复或回滚演练
- 你要把 D1 备份继续同步到对象存储
- 你要确认 `workflow_dispatch`、`git_ref`、`release/*`、告警 webhook 这些自动化细节

## 2. GitHub Actions 自动化

当前仓库已提供四个自动化 workflow：

- `.github/workflows/ci.yml`
  - 在 push / pull request / `workflow_dispatch` 时执行 `npm run ci:verify`
  - 依次覆盖 `npm run test:contract`、`npm run test:smoke`、`npm test`、`npm run test:unit`、`npm run typecheck`、`npm run build`、`npm run build:worker`
- `.github/workflows/deploy.yml`
  - `release/* -> staging`、`main -> production`
  - 也支持 `workflow_dispatch`
  - 可显式指定 `target_environment` 与 `git_ref`
  - 如果手动选择 `production`，必须从 `main` 分支上下文触发
- `.github/workflows/d1-backup.yml`
  - 每天 `03:15 UTC` 定时为 production 执行 full D1 导出
  - 手动触发时也可选择 `staging` / `production`、`full|schema|data` 与 artifact 保留天数
  - 如配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，会上传加密后的 `.enc` 与 `.sha256`
- `.github/workflows/failure-alert.yml`
  - 当 `CI`、`Deploy` 或 `D1 Backup` 失败时生成 failure summary
  - 如配置 `FAILURE_WEBHOOK_URL`，还会把失败信息推送到外部 webhook

部署 / 归档相关的 GitHub Secrets 与 vars：

- `CLOUDFLARE_API_TOKEN`
- `CLOUDFLARE_ACCOUNT_ID`
- `ADMIN_JWT_SECRET`
- `FAILURE_WEBHOOK_URL`
- `D1_BACKUP_ARCHIVE_PASSPHRASE`
- `D1_BACKUP_ARCHIVE_S3_URI`
- `D1_BACKUP_ARCHIVE_ENDPOINT_URL`
- `D1_BACKUP_ARCHIVE_AWS_REGION`
- `D1_BACKUP_ARCHIVE_SSE`
- `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
- `D1_BACKUP_ARCHIVE_STORAGE_CLASS`
- `D1_BACKUP_ARCHIVE_AWS_ACCESS_KEY_ID`

补充说明：

- workflow 当前默认使用 `npm ci`，并依赖已提交的 `package-lock.json`
- 当前 workflow 会先执行对应环境的远端 migration，再通过 `cloudflare/wrangler-action@v3` 在 `apps/worker` 目录下执行对应 `wrangler deploy` 命令，并上传 `ADMIN_JWT_SECRET` 到 Worker Secret
- 如果 `workflow_dispatch` 选择 `production`，但当前运行上下文不是 `main`，workflow 会显式失败
- 本地镜像 workflow 行为时，常用命令是 `npm run ci:verify`、`npm run build:worker:staging` 与 `npm run d1:restore:drill -- --file ./backups/d1/<backup.sql>`

## 3. D1 备份 / 恢复 SOP

建议在以下场景前先做一次远端 D1 备份：

- production 发版前，尤其是包含 migration、手工 SQL、批量导入或人工修数
- staging 验证涉及破坏性数据操作前
- 准备做数据回滚前，先导出当前“故障现场”

常用命令：

```bash
npm run backup:d1
npm run backup:d1:staging
npm run backup:d1 -- --scope schema
npm run backup:d1 -- --scope data
D1_BACKUP_ARCHIVE_PASSPHRASE=your-passphrase npm run backup:d1:encrypted
```

等价的原生命令示意：

```bash
mkdir -p backups/d1

npx wrangler d1 export subforge --remote --config ./wrangler.toml --output "./backups/d1/subforge-production-$(date +%Y%m%d-%H%M%S).sql"
npx wrangler d1 export subforge-staging --remote --env staging --config ./wrangler.toml --output "./backups/d1/subforge-staging-$(date +%Y%m%d-%H%M%S).sql"
npx wrangler d1 export subforge --remote --config ./wrangler.toml --no-data --output "./backups/d1/subforge-production-schema-$(date +%Y%m%d-%H%M%S).sql"
npx wrangler d1 export subforge --remote --config ./wrangler.toml --no-schema --output "./backups/d1/subforge-production-data-$(date +%Y%m%d-%H%M%S).sql"
```

关键说明：

- `wrangler d1 export <name>` 的参数是数据库名，不是绑定名
- 当前仓库 production / staging 分别对应 `subforge` 与 `subforge-staging`
- full export 同时包含 schema + data，因此应恢复到空库 / 新库
- `npm run backup:d1 -- --encrypt` 会生成 `.enc` 与对应 `.sha256`
- 解密可使用 `npm run backup:d1:decrypt -- --input <backup.sql.enc>`
- 当前仓库没有自动 down migration；如 schema 已损坏，仍需要依赖 Cloudflare Dashboard 和备份恢复完成回退

恢复建议：

- 先冻结发布，记录当前故障时间、commit、GitHub Actions run URL 与受影响环境
- 先把当前故障库再导出一份“事故快照”
- `整库恢复到空库 / 新库`：适合 full export
- `同 schema 数据回灌`：仅适合 `--no-schema` 备份，且目标库已经通过 migration 建好相同 schema

常用恢复命令：

```bash
npx wrangler d1 execute DB --remote --config ./wrangler.toml --file "./backups/d1/subforge-production-YYYYMMDD-HHMMSS.sql" -y
npx wrangler d1 execute DB --remote --env staging --config ./wrangler.toml --file "./backups/d1/subforge-staging-YYYYMMDD-HHMMSS.sql" -y
```

补充：

- 如果你使用的是 `--no-schema` 数据备份，先执行 `npm run db:migrations:apply` 或 `npm run db:migrations:apply:staging`
- 如需离线恢复演练，可执行 `npm run d1:restore:drill -- --file ./backups/d1/<backup.sql>`
- 如需对加密备份做演练，可直接传入 `.enc`

## 4. 对象存储归档

如果你准备让 GitHub Actions 自动把备份同步到对象存储，建议在目标 environment 中配置：

- `vars`
  - `D1_BACKUP_ARCHIVE_S3_URI`
  - `D1_BACKUP_ARCHIVE_ENDPOINT_URL`
  - `D1_BACKUP_ARCHIVE_AWS_REGION`
  - `D1_BACKUP_ARCHIVE_SSE`
  - `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
  - `D1_BACKUP_ARCHIVE_STORAGE_CLASS`
- `secrets`
  - `D1_BACKUP_ARCHIVE_AWS_ACCESS_KEY_ID`
  - `D1_BACKUP_ARCHIVE_AWS_SECRET_ACCESS_KEY`
  - `D1_BACKUP_ARCHIVE_AWS_SESSION_TOKEN`

workflow 实际会执行类似：

```bash
aws s3 cp backups/d1/ "s3://your-bucket/subforge-d1/<environment>/YYYY/MM/DD/gha-<run_id>/" --recursive --only-show-errors
```

约束：

- 如未配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，同步的是明文 SQL，此时必须配置 `D1_BACKUP_ARCHIVE_SSE`
- 如 `D1_BACKUP_ARCHIVE_SSE=aws:kms`，还必须配置 `D1_BACKUP_ARCHIVE_KMS_KEY_ID`
- 如已配置 `D1_BACKUP_ARCHIVE_PASSPHRASE`，对象存储收到的是 `.enc` 与 `.sha256`
- `D1_BACKUP_ARCHIVE_ENDPOINT_URL` 可用于 Cloudflare R2、MinIO 等 S3 兼容服务
- bucket lifecycle / prefix 留存策略需要在对象存储侧配置
