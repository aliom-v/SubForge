- **Forge**：锻造、生成

- 听起来像“把节点、规则、模板锻造成订阅”

- 也比较适合后面做成完整产品名

如果你想更偏中文语境，也可以叫：

- **订阅工坊**

- **订阅熔炉**

- **云订阅台**

- **节点工坊**

但如果是项目名、GitHub 仓库名、域名方向，我最推荐还是 **SubForge**。

---

# SubForge 项目方案

## 一、项目定位

SubForge 是一个部署在 Cloudflare 上的订阅管理平台，用来把你的节点、规则和模板统一管理，并输出为：

- Mihomo 订阅

- sing-box 订阅

它的目标不是去运行代理核心，而是做：

- 后台管理

- 配置存储

- 订阅生成

- 规则分发

- 同步更新

- 用户 token 分发

也就是：  
**像机场那样提供订阅 URL，但管理方式更轻、更适合你自己维护。**

---

## 二、为什么这是最推荐的方案

因为你当前的需求非常明确：

- 你已经有配置文件

- 想做成“像机场一样 URL 拉订阅”

- 想有前端后台去增删改

- 想支持后续继续扩展

- 希望能部署在 Cloudflare

- 最好一键部署

那最稳的路线就是：

### 前端后台

负责增删改查

### Worker API

负责生成订阅和同步配置

### D1

负责存结构化数据

### KV

负责缓存最终订阅

### Cron

负责定时同步

这套结构的优点是：

- 不需要自建传统后端服务器

- 不需要一直在线的常驻程序

- 很适合做“订阅分发型服务”

- 后续扩展也自然

## 五、最推荐的架构

## 1. 前端后台

一个管理页面，负责：

- 新增节点

- 删除节点

- 修改节点参数

- 新建用户

- 分配 token

- 查看订阅地址

- 编辑 Mihomo 模板

- 编辑 sing-box 模板

- 配置规则源

- 手动点击同步

---

## 2. Worker API

后端接口负责：

- 节点 CRUD

- 用户创建 / 编辑 / token 管理 / 节点绑定

- 模板创建 / 更新 / 默认模板切换

- 规则源创建 / 更新 / 手动同步

- 生成最终订阅

- 清理缓存

- 返回配置预览

---

## 3. D1

存这些数据：

- users

- nodes

- templates

- rulesources

- user_node_map

- sync_logs

---

## 4. KV

缓存这些内容：

- 某个 token 对应的 Mihomo 订阅

- 某个 token 对应的 sing-box 订阅

- 最新模板快照

- 最近一次规则同步结果

---

## 5. Cron

做这些定时任务：

- 定时同步启用的远程规则源

- 刷新规则快照并按需失效缓存

- 定时清理过期用户缓存

# 六、配置怎么维护

这部分是整个项目最关键的设计。

我最推荐你把配置拆成 3 层。

## 第一层：节点层

只存节点原始信息：

- 协议

- 地址

- 端口

- uuid / password

- tls

- sni

- 标签

- 分组

这一层不要写规则，也不要写完整最终配置。

---

## 第二层：模板层

分别维护：

- Mihomo 模板

- sing-box 模板

模板里放：

- DNS

- 出站组

- 策略组

- 路由骨架

- 规则引用入口

- 通用配置

---

## 第三层：规则层

单独维护规则源：

- proxy

- direct

- reject

- media

- ai

后端渲染时把这些规则源地址注入模板。

这样以后：

- 加节点，只改节点

- 改分流，只改规则

- 改整体结构，只改模板

这是后续最省事的方式。

- 注入节点和规则

- 生成 YAML

# 十二、一键部署方案

SubForge 最适合做成 **GitHub 模板仓库 + Wrangler 部署**。

## 用户部署步骤

### 1. Fork 项目

创建自己的仓库副本。

### 2. 登录 Cloudflare

npx wrangler login

### 3. 创建 D1

npx wrangler d1 create subforge

### 4. 创建 KV

npx wrangler kv namespace create SUB_CACHE

### 5. 写入 wrangler.toml

绑定：

- D1

- KV

- 环境变量

### 6. 执行数据库迁移

npx wrangler d1 migrations apply subforge

### 7. 部署

npm run deploy

这样已经很接近一键部署了。

后面你还可以补：

- GitHub Actions 自动化部署

- 首次初始化脚本

- Demo 数据导入脚本

# 十三、Cloudflare 不支持或不适合的地方

这个项目设计时必须避开这些。

## 1. 不适合常驻进程

不能把 Cloudflare 当：

- 一直在线的代理程序

- 持续测速器

- 死循环同步器

- 守护进程

所以代理核心必须还在你的 VPS 上。

---

## 2. 不适合重计算

不适合：

- 大规模复杂转换

- 超大配置实时运算

- 很重的批处理

所以生成逻辑要尽量轻量、模板化。

---

## 3. 不推荐做动态脚本引擎

不要把用户自定义 JS 脚本作为核心功能。

例如这类：

- eval 风格脚本扩展

- 运行任意 JS 改配置

- 复杂运行时脚本注入

这些在 Cloudflare 上不是最稳的路线。

**推荐永远用：**

- 显式模板

- 明确配置项

- 明确规则源
